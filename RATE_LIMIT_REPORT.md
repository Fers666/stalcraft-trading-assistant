## 2. Реализация в коде

### Алгоритм Token Bucket

```python
# rate_limiter.py:34-67 (Lua-скрипт)
CAPACITY     = 100 токенов
REFILL_RATE  = 100/60 ≈ 1.667 токена/сек

tokens = min(CAPACITY, tokens + elapsed × REFILL_RATE)

if tokens ≥ needed:
    tokens -= needed → OK
else:
    wait = (needed − tokens) / REFILL_RATE → ждём
```

### Хранилище: Redis

| Параметр | Значение |
|----------|----------|
| **Ключ** | `stalcraft:rate_limit` |
| **Поля** | `tokens` (остаток), `last_refill` (timestamp) |
| **TTL** | 120 секунд |
| **Атомарность** | Lua-скрипт (проверка + списание в одной операции) |

### Fallback: In-Memory Bucket

Если Redis недоступен, используется в-памятный bucket (одного процесса, без гарантий при нескольких воркерах Celery).

## 3. Текущее использование токенов

### 📈 Нагрузка по задачам

| Задача | Расписание | Кол-во запросов | Стоимость/запрос | Токенов/цикл | Токенов/минуту |
|--------|-----------|-----------------|------------------|--------------|----------------|
| `collect_all_active_lots` | каждые 20 сек | 5 предметов | 2 (LOTS) | 10 | ~30 |
| `run_global_feed_batch` | каждую минуту | 12 предметов | 2 (LOTS) | 24 | 24 |
| `collect_all_history` | раз в час (мин. 0) | ~16 товаров | 2 (HISTORY) | 32 | ~0.5 (распред.) |
| **Итого текущее** | | | | | **~54-55** |

**Расчёты:**
- `collect_all_active_lots`:
  - LOTS_PER_RUN = 5 (`collectors.py:13`)
  - LOTS_REFRESH_INTERVAL = 20 сек
  - 60 сек / 20 сек = 3 запуска в минуту
  - 5 предметов × 2 токена × 3 запуска = **30 токенов/мин**

- `run_global_feed_batch`:
  - BATCH_SIZE = 12 (`global_scanner.py:22`)
  - Запускается каждую минуту
  - 12 предметов × 2 токена = **24 токена/мин**

- `collect_all_history`:
  - Раз в час, ~16 товаров
  - 16 × 2 = 32 токена / 60 мин = **~0.5 токена/мин**

## 4. Резервный потенциал

### 🎯 Текущее использование vs. Лимит

```
Используется:     54-55 токенов/мин (54-55% от лимита)
Доступно:         100 токенов/мин
Резерв:           ~45-46 токенов/мин (45-46% неиспользовано)
```

### 💡 Потенциальные оптимизации

1. **Увеличить LOTS_PER_RUN** (сейчас 5, максимум ~15-16)
   - При LOTS_PER_RUN = 15: 15 × 2 × 3 = **90 токенов/мин**
   - Ограничение: не рекомендуется использовать более 90% лимита (буфер на случай спайков)

2. **Уменьшить LOTS_REFRESH_INTERVAL** (сейчас 20 сек)
   - Например, 15 сек: 4 запуска/мин вместо 3
   - Требует пропорционального уменьшения LOTS_PER_RUN

3. **Уменьшить BATCH_SIZE** в run_global_feed_batch (если нужна экономия)
   - Например, 8 предметов/мин = 16 токенов/мин (минус 8 токенов/мин)

## 5. Сценарии и рекомендации

### 📌 Сценарий 1: Текущая нагрузка (консервативный)

- **Статус:** ✅ Безопасен
- **Рекомендация:** Оставить как есть (54-55% от лимита)
- **Причина:** 45% резерв достаточен для спайков и новых задач

### 📌 Сценарий 2: Оптимизированная нагрузка (агрессивный)

- **Цель:** Максимизировать обновление данных
- **Действия:**
  1. Увеличить LOTS_PER_RUN с 5 до 10:
     - `collectors.py:13`: `LOTS_PER_RUN = 10`
     - Нагрузка: 10 × 2 × 3 = 60 токенов/мин
  2. Уменьшить LOTS_REFRESH_INTERVAL с 20 до 15 сек:
     - `collectors.py:12`: `LOTS_REFRESH_INTERVAL = 15`
     - Результат: 10 × 2 × 4 = 80 токенов/мин
  3. Итого нагрузка: 80 + 24 (глобальный скан) = **104 токена/мин**
     - ⚠️ **Превышает лимит на 4%!** Не рекомендуется.

- **Безопасная оптимизация:**
  1. LOTS_PER_RUN = 8: 8 × 2 × 3 = 48 токенов/мин
  2. BATCH_SIZE = 10: 10 × 2 = 20 токенов/мин
  3. Итого: 48 + 20 = **68 токенов/мин** (68% использования, 32% резерв)

### 📌 Сценарий 3: Сбалансированный (рекомендуемый)

- **Цель:** Увеличить скорость обновления с небольшим запасом
- **Действия:**
  1. LOTS_PER_RUN = 7: 7 × 2 × 3 = 42 токена/мин
  2. BATCH_SIZE = 13: 13 × 2 = 26 токенов/мин
  3. Итого: 42 + 26 = **68 токенов/мин** (68% использования)
- **Эффект:** На 14 токенов больше, ~26% буфер на спайки

## 6. Мониторинг

```python
from app.core.rate_limiter import rate_limiter

status = await rate_limiter.get_status()
# Вывод:
# {
#   "tokens_available": 85.3,
#   "capacity": 100,
#   "refill_rate_per_min": 100,
#   "source": "redis"  # или "fallback"
# }
```

**Логирование:**
- `rate_limiter.py:138` — info log при ожидании токенов
- `StalcraftClient._request()` — debug logs всех API запросов

**Метрики для Grafana:**
- `stalcraft:rate_limit` — значение в Redis (текущий остаток)
- Логи Celery worker — время ожидания rate limit
