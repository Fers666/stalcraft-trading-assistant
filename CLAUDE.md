# SC Trading — Контекст для Claude

## Блок 1: Управление контекстом

Одна задача = один чат. После завершения → /clear, открыть новый чат.  
История длинная в середине задачи → /compact, продолжить.  
В конце задачи (Шаг 6): сообщи «Задача выполнена. /clear».

## Блок 2: Алгоритм работы

| Задача | Читать |
|--------|--------|
| Что делать следующим | docs/NOTES.md |
| Формулы, расчёты | docs/BUSINESS_LOGIC.md |
| Сервисы, Celery задачи | docs/SERVICES.md |
| Таблицы БД, миграции | docs/DATABASE.md |
| Архитектура сбора данных | docs/ARCHITECTURE.md |
| Деплой, сервер, инфраструктура | docs/DEPLOY.md |

1. Читай `docs/NOTES.md` → первая `[ ]` задача
2. `researcher` — исследует код/docs, пишет ТЗ в `docs/tasks/<slug>.md` → **жди подтверждения**
3. `backend-dev` / `frontend-dev` / `designer` — реализуют по ТЗ → **жди подтверждения** перед коммитом
4. `tech-writer` — обновляет docs/ (только diff): `[ ]` → `[x]`, метод/таблица/формула
5. Предложи (не делай автоматически): «Хочешь протестировать? Могу вызвать `qa-tester`»
6. Предложи (не делай автоматически): «Хочешь задеплоить? Могу вызвать `deploy`»
7. Предложи (не делай автоматически): `git commit` + `git push` → `/clear`

## Блок 3: Правила коммуникации

До действия: «Планирую: [действие]. Затрону: [файлы]. Подтверждаешь?»  
Без подтверждения: не изменяй файлы, не коммить, не применяй миграции.  
После: «Сделано: [факт]. Изменено: [список файлов]»

## Блок 4: Краткий контекст

Аналитика аукциона Stalcraft X — находить выгодные лоты для перепродажи.

**Стек:** FastAPI + SQLAlchemy 2.0 async + PostgreSQL 16 (UTC+3) + Celery + Redis  
React 18 + Vite + MUI (gold: #D9AF37, bg: #080808, font: Rajdhani)  
Docker Compose. Данные глобальные (user_id=NULL), персонализация на уровне запроса.

**API Rate Limit:** 400 запросов/мин (verified experimentally 2026-06-07, NOT документация 100 токенов!)
- /lots = 2, /history = 2, /emission = 1 (Request-based, не token-based)
- Текущее: 54.5 запросов/мин (13.6% от лимита) — БЕЗОПАСНО
- Reserve: 345.5 запросов/мин (86.4%)
- **При оптимизации:** спрашивай разрешение перед изменением LOTS_PER_RUN / BATCH_SIZE / REFRESH_INTERVAL
- Token Bucket → Redis (Lua). Fallback: in-memory.

Детали → читай docs/

## Блок 5: Специализированные агенты

Для объёмных задач — делегировать subagent'у (Agent tool):

| Агент | Когда вызывать |
|-------|---------------|
| `researcher` | ПЕРВЫМ для любой нетривиальной задачи — исследует, планирует, пишет ТЗ |
| `backend-dev` | Реализация: FastAPI, SQLAlchemy, Celery, Alembic |
| `frontend-dev` | Реализация: React, MUI, Zustand |
| `designer` | Макеты, прототипы в design/, дизайн-спека |
| `tech-writer` | После задачи: обновление docs/, CHANGELOG.md, CLAUDE.md |
| `qa-tester` | Предлагать после реализации — анализ багов, проверка API/логики |
| `deploy` | Предлагать после тестирования — деплой на VPS |
| `security` | Перед деплоем или по запросу — OWASP ревью |

Агенты следуют правилам коммуникации из Блока 3 (план → подтверждение → выполнение).
