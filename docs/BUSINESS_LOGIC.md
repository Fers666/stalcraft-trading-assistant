# Бизнес-логика Stalcraft Trading Assistant

## Обзор системы

Приложение анализирует аукцион игры Stalcraft X. Каждые 5 минут собираются активные лоты, раз в час — история продаж. На основе этих данных рассчитываются рекомендации по покупке и прогнозы времени продажи.

**Основной цикл:**
1. Celery воркер собирает лоты (каждые 5 мин) и историю продаж (раз в час)
2. Analytics сервис пересчитывает статистику (раз в час, на 5-й минуте)
3. FastAPI выдаёт рекомендации
4. Уведомления отправляются в Telegram / браузер

---

## 1. Расчёт выгодности покупки

```
expected_revenue = avg_sell_price_7d × (1 − 0.05)   // минус 5% комиссия
profit_per_unit  = expected_revenue − lot.price_per_unit
profit_percent   = (profit_per_unit / lot.price_per_unit) × 100

confidence = min(1.0, sales_volume_7d / 100)
score      = profit_percent × confidence
```

**Константы:**
- Комиссия продажи: **5%**
- Минимальная маржа по умолчанию: **10%** (пользователь может изменить)
- Уверенность = 1.0 при 100+ продажах за неделю

**Условие показа рекомендации:** `profit_percent ≥ user.min_profit_margin_percent`

## 2. Риск-анализ (волатильность)

```
volatility = (stdev(prices_7d) / mean(prices_7d)) × 100
```

| Уровень | Условие |
|---------|---------|
| LOW | volatility ≤ 15% |
| MEDIUM | 15% < volatility ≤ 30% |
| HIGH | volatility > 30% |

## 3. Медиана цены (median_price_7d)

Показывает типичную цену продажи товара за последние 7 дней.

**Почему медиана, а не среднее:**
Среднее искажается аномальными сделками (ошибочная цена, другой товар с похожим названием).
Медиана устойчива к выбросам — один странный лот не сдвигает результат.

```
Пример:
  Продажи: [3 900 000, 3 950 000, 4 000 000, 4 050 000, 9 999 999]
  Среднее  = 5 179 999  ← сдвинуто аномалией
  Медиана  = 4 000 000  ← отражает реальный рынок
```

**Формула:**
```
prices_7d     = [sale.price_per_unit for sale in sales_history
                 where sale_time >= now − 7 дней]

median_7d     = statistics.median(prices_7d)
```

**Переменные:**
- `prices_7d` — список цен за единицу товара за 7 дней
- `sale.price_per_unit = sale.total_price // sale.amount` — цена за штуку (целочисленное деление)
- `sale.total_price` — итоговая сумма сделки из API `/history`
- `sale.amount` — количество единиц в сделке

**Что важно знать:**
- `median_7d` — это цена **покупателя**. Продавец получает `median_7d × 0.95` (минус 5% комиссия)
- Комиссия уже учтена в расчёте вариантов продажи (раздел 4)
- Если за 7 дней не было продаж — поле равно `null`, карточка показывает `—`

**Используется:**
- Как базовая цена для вариантов "Нормально" и "Выгодно" в прогнозе продажи
- Для расчёта волатильности (раздел 2)

## 4. Прогноз времени продажи

Для каждого товара генерируются **3 ценовые стратегии**:

`ref = current_min_liquid_price or median_7d` — база отсчёта.

> **Свежесть данных:** `sell_options` хранятся в `market_statistics` (JSONB) и пересчитываются раз в час задачей `calculate_all_market_stats`. Чтобы избежать ложных "выгодных лотов" при быстром падении рынка, эндпоинт `GET /monitoring/item/{id}` при каждом запросе берёт последний снапшот `CollectedData` и пересчитывает `sell_options` "на лету". Задержка реакции: ~5 мин (интервал коллектора). Сохранённые значения — только fallback.

> **Граничный случай:** если `prices=[]` (нет реальных продаж за 7 дней для данного фильтра), `pricing.make_sell_options` возвращает `None`. `compute_signals_for_entry` и `GET /monitoring/item` обрабатывают `None` явно: сигнал не строится, UI показывает "Нет данных о продажах".

| Стратегия | Формула цены | Смысл |
|-----------|---|---|
| Быстро | `ref × 0.97` | Ниже рынка → быстрая продажа |
| Нормально | `ref × 1.00` | По рынку |
| Выгодно | `ref × 1.05` | Выше рынка за ожидание |

`net_price_per_unit = price × 0.95` — цена после комиссии 5% (показывается в UI).

### Алгоритм расчёта времени (3 уровня)

Уровень точности определяется **покрытием** — какой процент продаж за 30 дней
имеет восстановленный `lot_start` (время выставления лота):

```
coverage = matched_count / total_sales_30d × 100%

matched_count    — продажи, для которых найден lot_start через матчинг снэпшот→история
total_sales_30d  — все продажи за 30 дней из API /history
```

**Уровень 1 — Высокая точность** (`coverage ≥ 30%` И `matched_count ≥ 10`):

Используем реальные пары `(price_per_unit, time_on_market)`.
По цене каждой стратегии берём 5 ближайших точек → среднее их времени жизни.

**Уровень 2 — Средняя точность** (`coverage 10–30%` И `matched_count ≥ 3`):

Данных достаточно для направления, но мало для точной интерполяции.
```
avg_time = среднее time_on_market по всем matched точкам

fast:    avg_time × 0.4   // ниже рынка → продаётся быстрее среднего
normal:  avg_time × 1.0   // рыночная цена → среднее время
premium: avg_time × 2.5   // выше рынка → дольше ждать
```

**Уровень 3 — Оценка по активности рынка** (`coverage < 10%`):

Нет достаточных реальных данных. Используем `sales_volume_7d` (продаж за 7 дней)
как косвенный показатель скорости рынка.

```
sales_per_day = sales_volume_7d / 7

sales_per_day ≥ 5       → активный рынок:    fast=2ч  / normal=8ч   / premium=24ч
sales_per_day 1–5       → умеренный рынок:   fast=8ч  / normal=24ч  / premium=72ч
sales_per_day 0.14–1    → редкий (~1/неделю): fast=24ч / normal=72ч  / premium=168ч
sales_per_day < 0.14    → очень редкий:       fast=72ч / normal=168ч / premium=336ч
```

**Переменные confidence (отображаются в UI):**
- `high`   — покрытие ≥30%, ≥10 точек с lot_start
- `medium` — покрытие 10–30%, ≥3 точки
- `low`    — покрытие <10%, оценка по объёму

## 5. Лучшее время: продавать и покупать

В карточке Избранного отображаются два показателя с переключателем **Неделя / Сегодня**:

- **Продавать** — час и день когда выгоднее всего выставить лот
- **Покупать** — час и день когда цены на аукционе исторически наименьшие

Переключатель влияет только на этот блок:
- **Неделя** — лучший день+час по всем данным за 30 дней
- **Сегодня** — лучший час именно для текущего дня недели (если данных нет — fallback на "Неделя")

### Лучший час продажи (best_sell_hour / sell_hours_by_day)

Источник: `sales_history` за 30 дней.

**Взвешенный скор (60% цена + 40% объём):**
```
Для каждого часа h:
  avg_price(h) = mean(price_per_unit) всех продаж в этот час
  volume(h)    = количество продаж в этот час

  price_norm(h)  = avg_price(h) / max(avg_price по всем часам)  → [0, 1]
  volume_norm(h) = volume(h) / max(volume по всем часам)        → [0, 1]

  score(h) = price_norm(h) × 0.6 + volume_norm(h) × 0.4

best_sell_hour = час с максимальным score
```

**Почему 60/40:** нас интересует не просто когда продаётся больше всего, а когда покупатели платят БОЛЬШЕ при достаточно активном рынке.

**По дням:** для режима "Сегодня" рассчитывается `sell_hours_by_day`:
```
sell_hours_by_day = {"Monday": 20, "Tuesday": 19, "Wednesday": 21, ...}
```
Тот же взвешенный скор, но только по продажам конкретного дня недели.

Переменные:
- `WEIGHT_PRICE = 0.6` — вес ценового фактора
- `WEIGHT_VOLUME = 0.4` — вес объёмного фактора
- Минимум данных: `MIN_SALES_FOR_STATS = 3` продажи за 30 дней

### Лучший час покупки (best_buy_hour / buy_hours_by_day)

Источник: `collected_data` — снэпшоты лотов каждые 5 минут за 30 дней.

```
Для каждого часа h:
  prices(h) = [best_liquid_price_per_unit из каждого снэпшота в этот час]

  avg_min_price(h) = mean(prices(h))

best_buy_hour = час с минимальным avg_min_price
```

`best_liquid_price_per_unit` — минимальная цена среди ликвидных лотов (с остатком ≥ 2ч).
В это время продавцы исторически выставляют товар дешевле → выгодное время для покупки.

**По дням:** аналогично → `buy_hours_by_day = {"Monday": 2, "Tuesday": 3, ...}`

Минимум данных: 6 снэпшотов (~30 минут).

## 6. Источник данных о продажах

Единственный источник — **Stalcraft API `/history`**, который возвращает реальные завершённые сделки.
Данные 100% достоверны: факт продажи, цена и время подтверждены самим аукционом.

```
GET /{region}/auction/{item_id}/history
→ "prices": [{ "amount": 1, "price": 3 699 990, "time": "2026-05-29T16:55:32Z" }]
```

Переменные записи:
- `amount`  — количество единиц товара в одной сделке
- `price`   — итоговая сумма всей сделки (не за штуку)
- `time`    — момент совершения сделки

В `sales_history` сохраняется:
- `price_per_unit = price // amount` — цена за единицу
- `total_price = price` — итог сделки

### Восстановление времени нахождения на рынке и качества/заточки (lot_start, qlt, ptn)

API `/history` не сообщает ни когда лот был **выставлен**, ни его **качество/заточку** — только факт продажи, цену и время.

При каждом сборе истории система пытается найти соответствующий лот в снэпшотах (`collected_data.raw_lots`) через функцию `find_lot_info`.

**Алгоритм матчинга (поиск лота по продаже):**
```
Условия совпадения:
  lot.buyoutPrice == sale.price   — одинаковая сумма сделки
  lot.amount == sale.amount       — одинаковое количество
  lot.endTime > sale_time         — лот куплен до истечения (не истёк сам)

Лот должен:
  → присутствовать в снэпшоте ДО sale_time
  → отсутствовать в снэпшоте ПОСЛЕ sale_time

При нескольких кандидатах (одинаковые лоты):
  → берём с наиболее ранним startTime среди исчезнувших
```

Если лот найден — в `sales_history.additional_info` сохраняется:
- `lot_start` — время выставления лота → `time_on_market = sale_time - lot_start`
- `qlt` — уровень качества (0–5, только у артефактов)
- `ptn` — точность артефакта: `0` / отсутствие = "Не точёный"; `1–15` = уровень качества

Окно матчинга: последние **200 снэпшотов** (~1.7 часа при интервале 20 сек), что перекрывает hourly-цикл сбора истории.

**Текущее покрытие:** 2–6% продаж имеют `qlt`/`ptn` — покрытие растёт по мере работы системы.

### Статистика артефактов с фильтром quality+enchant

Когда карточка Избранного имеет выбранное качество и/или точность, `/monitoring/item` и `/monitoring/sales-chart` фильтруют `SalesHistory` по `additional_info->>'qlt'` и `additional_info->>'ptn'`.

Специальный случай: `enchant_filter=0` ("Не точёный") → SQL: `ptn IS NULL OR ptn = '0'` (покрывает старые записи без ptn и новые с ptn=0).

**Принцип:** все расчёты и предложения строятся **только на реальных совершённых продажах**. Snapshot-данные (цены выставленных лотов) намеренно не используются как замена — они не отражают реальные сделки и делают предложения нерелевантными.

При недостатке подходящих продаж карточка честно показывает "Нет данных о продажах".

### Ликвидные vs неликвидные лоты

При каждом снэпшоте лоты разделяются:
```
liquid_lots:   lot.endTime > now + 2 часа  // ликвидные — цена актуальна
expiring_lots: lot.endTime ≤ now + 2 часа  // истекающие — никто не купил по этой цене
```

Переменная `EXPIRY_THRESHOLD_HOURS = 2` — порог в часах.

`best_liquid_price_per_unit` — минимальная цена только среди ликвидных лотов.
Используется как базовая цена для варианта "Быстро" в прогнозе.

## 7. Подбор пачки (Batch Matcher)

Жадный алгоритм: набрать `target_quantity` штук по минимальной цене.

```
sorted_lots = сортировка лотов по price_per_unit (возрастание)
remaining   = target_quantity

for lot in sorted_lots:
    take = min(lot.amount, remaining)
    remaining -= take
    if remaining == 0: break

if remaining > 0: return None  // не хватает товара
```

## 9. Rate Limiting (Request-Based)

**⚠️ ОБНОВЛЕНО 2026-06-07: Реальные лимиты Stalcraft API (экспериментально проверены)**

**Лимиты:** 400 запросов / минута (НЕ 100 токенов!)

| Запрос | Стоимость | Примечание |
|--------|----------|-----------|
| `/auction/{id}/lots` | 2 запроса | Активные лоты |
| `/auction/{id}/history` | 2 запроса | История продаж |
| `/emission` | 1 запрос | Статус радиации |

**Система отслеживания:**
- API возвращает headers: `x-ratelimit-limit`, `x-ratelimit-remaining`, `x-ratelimit-reset`
- Period: ровно 60 секунд
- Reset timestamp в миллисекундах (Unix time × 1000)

**Алгоритм:**
```
CAPACITY     = 400 запросов (проверено экспериментально)
REFILL_RATE  = 400/60 ≈ 6.667 запросов/сек

tokens = min(CAPACITY, tokens + elapsed × REFILL_RATE)

if tokens ≥ needed:
    tokens -= needed → OK
else:
    wait = (needed − tokens) / REFILL_RATE → ждём
```

Реализован через атомарный Lua-скрипт в Redis.

## 11. Расписание задач Celery

| Задача | Расписание | Описание |
|--------|-----------|---------|
| `collect_all_active_lots` | каждые 20 сек | 1 предмет за запуск (очередь по `last_successful_check ASC`) |
| `collect_all_history` | раз в час (мин. 0) | Сбор истории продаж + матчинг lot_start/qlt/ptn (дедуп по (item_id, region)) |
| `calculate_all_market_stats` | раз в час (мин. 5) | Пересчёт market_statistics и sell_options |
| `run_global_feed_batch` | каждую минуту | 12 предметов вне watchlist (~3 ч полный цикл) |
| `delete_old_data` | ежедневно 03:00 | Удаление данных старше 120 дней |
| `sweep_expired_tiers` | ежедневно 03:30 | Понижение до `base` пользователей с истёкшим `tier_expires_at` (между cleanup и часовой статистикой; см. раздел 17) |
| `force_refresh_all_history` | вручную (Admin) | Принудительный пересбор истории + пересчёт статистики |

### Архитектура сбора лотов (`collect_all_active_lots`)

**Дедупликация по (item_id, region):** если 100 пользователей следят за одним товаром в одном регионе — делается ровно 1 API-запрос. Обновление `last_successful_check` распространяется на все записи watchlist с совпадающей парой.

**Очередь по давности:** записи сортируются `last_successful_check ASC NULLS FIRST` — самые устаревшие обрабатываются первыми. Новые записи (NULL) всегда идут в начало.

**Анти-всплесковый механизм:** `LOTS_PER_RUN = 1` — один предмет за запуск. После рестарта воркера все записи одновременно устаревают, но ограничение предотвращает спайк из десятков одновременных API-запросов.

**Константы:**
```
LOTS_REFRESH_INTERVAL = 20 сек   — интервал обновления одного предмета
LOTS_PER_RUN          = 1        — предметов за один запуск задачи
LOTS_REQUEST_DELAY    = 2 сек    — пауза между запросами (при LOTS_PER_RUN > 1)
```

**Redis-кэш:** обновляется сразу после получения ответа от Stalcraft API, до коммита в БД. Это гарантирует, что `GET /lots/{id}` отдаёт свежие данные без задержки на запись в PostgreSQL.

### Триггерный сбор при добавлении в watchlist

При добавлении нового предмета в избранное API немедленно запускает две задачи без ожидания планировщика:
- `collect_single_item` — снэпшот активных лотов (throttle Redis: не чаще раза в 2 минуты)
- `collect_history_single` — начальная история продаж

### Ретроактивное обогащение qlt/ptn

`collect_all_history` при каждом запуске не только добавляет новые продажи, но и обогащает **уже существующие** записи, у которых отсутствует `qlt`:

```
Если запись в sales_history не имеет additional_info.qlt:
  → пробуем сматчить с текущими снэпшотами
  → если лот найден — UPDATE additional_info с qlt + ptn + lot_start
```

Это позволяет ретроактивно восстанавливать качество/заточку для продаж, собранных до того как в API-запрос был добавлен параметр `additional=true`.

### Дедупликация sales_history

`collect_all_history` (как и `force_refresh_all_history`) дедуплицирует записи watchlist по `(item_id, region)` перед сбором — один API-запрос `/history` на пару, независимо от того, сколько пользователей следят за предметом.

Внутри `_collect_history_for_item` каждая продажа из ответа `/history` сверяется с уже сохранёнными записями (за последние 120 дней) по ключу `(sale_time ± 1 сек, total_price, amount)`. Если совпадение найдено — продажа считается уже собранной (при отсутствии `qlt` дополнительно пробуется ретроактивный матчинг лота). Если нет — продажа сохраняется через `INSERT ... ON CONFLICT DO NOTHING`, а сразу после вставки добавляется во внутренний индекс прохода — это защищает от дублей, если API возвращает одну и ту же продажу два раза в одном ответе `/history`.

На уровне БД действует уникальный индекс `uq_sales_history_sale (item_id, region, sale_time, total_price, amount)` — последняя линия защиты от дублей при гонках между параллельными задачами (`worker_concurrency=2`).

## 15. Telegram Bot — уведомления о выгодных лотах

Сервис `telegram_bot/bot.py` — отдельный Docker-контейнер (python-telegram-bot v21).

**Привязка аккаунта:**
1. Пользователь нажимает «Получить код привязки» в Настройках
2. Backend генерирует 6-значный код, сохраняет `{tg_link:CODE → user_id}` в Redis (TTL 10 мин)
3. Пользователь отправляет боту: `/link XXXXXX`
4. Бот находит `user_id` по коду, сохраняет `telegram_chat_id` в таблицу `users`

**Логика уведомлений (15 сек polling Redis-ключей `signals:*`):**

```
Коллектор раз в ~20-60с (после сбора снэпшота) вызывает
profitable_lots.compute_signals_for_entry для каждой watchlist-записи
и пишет результат в Redis (signals:{user_id}:{item_id}:{region}:{qf}:{ef}, TTL 300с).

Для каждого пользователя с telegram_chat_id:
  Если user_settings.notify_telegram = true И (is_admin ИЛИ тариф разрешает telegram_notifications):
    (тарифный гейт — Phase 0, см. раздел 17; НЕ влияет на саму привязку аккаунта,
     только на отправку проактивных уведомлений — привязка доступна всем тарифам)
    Для каждой записи user_watchlist (is_active=True):
      1. Читаем signals:* из Redis (если ключ есть — сигнал свежий, иначе пропуск)
      2. ref/trend — pricing.compute_reference (см. ниже), profit считается от
         тира "fast" с поправкой на пачку и риск-маржу (pricing.evaluate_lot_profit)
      3. Для каждого профитного лота из signals["lots"] (уже отсортированы по
         profit_per_hour):
         - Dedup: Redis key tg_sent:{user_id}:{item_id}:{region}:{qf}:{ef}:{startTime}
           TTL = 48ч (один лот — одно уведомление за 48 ч)
         - Отправить HTML-сообщение в telegram_chat_id
```

**Расчёт `ref` и `trend` (`pricing.compute_reference`):**
`ref` приоритетно берётся из `median_price_7d` (7-д медиана из `market_statistics`,
для записей с quality/enchant фильтром — медиана `SalesHistory` за 7д по тому же
qlt/ptn; если истории нет — `current_min` как fallback). Дополнительно сравнивается
с медианой ТЕКУЩЕГО снэпшота (`median_now`):
- `median_now / median_hist < 0.75` → `trend="falling"` (рынок "просел"), `ref`
  консервативно корректируется вниз: `max(median_now, median_hist × 0.75)`
- `median_now / median_hist > 1/0.75` → `trend="rising"`
- иначе → `trend="stable"`

**Почему ref ≠ current_min:** использование `ref = current_min_liquid` (текущий
минимум рынка) делало условие выгодности математически невозможным — ref сам
является минимумом, все лоты имеют `price ≥ ref`. Медиана за 7 дней (с
trend-guard на случай резкого падения рынка) позволяет находить лоты, выставленные
ниже исторического уровня.

**Профит лота (`pricing.evaluate_lot_profit`):**
- Цена продажи берётся из тира **"fast"** (`ref × 0.97`), после вычета комиссии 5%
- Если для размера пачки лота есть ≥3 реальных продажи в `market_statistics.batch_stats`
  (по бакетам `pricing.BATCH_BUCKETS`) — цена продажи корректируется пропорционально
  медианной цене такой пачки
- `required_margin = min_profit_margin_pct × RISK_MARGIN_MULT[risk]` — при `risk="high"`
  (волатильность >30%) требуется в 1.6× больше маржи, чем при `risk="low"` (≤15%)
- Лот считается выгодным, если `profit_pct >= required_margin`

**Содержимое уведомления:**
- Название предмета
- Качество (⭐) и заточка (⚡) — из фильтров watchlist
- Цена выкупа (💵)
- Прогноз продажи в 3 срезах: Быстро / Нормально / Выгодно с чистой прибылью и % (📈)
- Доходность ₽/час на тарифе "Быстро" (⏱)
- Предупреждение, если `trend == "falling"` (рынок ниже недельной медианы)
- Предупреждение, если `saturation_ratio > 1` (много похожих профитных лотов сразу — рынок может не успеть их переварить)
- Продаж за 7д (📦) и волатильность за 7д (📉)
- Точность прогноза: высокая / средняя / мало данных (🎯)

**Команды бота:**
- `/start` — инструкция по привязке
- `/link CODE` — привязать аккаунт по коду
- `/status` — статус привязки + кол-во предметов в вотчлисте
- `/stop` — отвязать аккаунт (обнуляет telegram_chat_id)

## 16. Краткое резюме алгоритмов

1. **Выгодность** = (ref × 0.97 × 0.95 − цена_лота) / цена_лота × 100% ≥ min_margin × RISK_MARGIN_MULT[risk], где `ref` = `median_price_7d` с trend-guard по медиане текущего снэпшота (`pricing.compute_reference`)
2. **Уверенность** = min(1.0, продаж_за_7д / 100)
3. **Доходность** = `profit_per_hour` = профит / прогнозируемое время продажи (тир "fast") — основной критерий ранжирования сигналов
4. **Риск** = волатильность: >30% (high), >15% (medium), ≤15% (low) → множитель требуемой маржи 1.6 / 1.3 / 1.0
5. **Время продажи** = интерполяция по реальным выкупам (или эвристика по продажам/день)
6. **Batch matching** = жадный алгоритм по цене до target_quantity; для оценки профита — поправка цены продажи по `market_statistics.batch_stats` при ≥3 продажах в том же бакете объёма
7. **Snapshot matching** = лот в снэпшоте ДО продажи и отсутствует ПОСЛЕ → восстанавливаем lot_start → time_on_market
8. **Rate limit** = Token Bucket: 400 запросов/мин (период 60 сек), см. §9
9. **Saturation** = `total_profitable_amount / (sales_volume_7d / 7)` — много профитных лотов сразу относительно недельного объёма продаж → рынок может не успеть их переварить
10. **Demand bulk_spike** = резкий рост доли объёма продаж в пачках ≥10 шт за 24ч против базовой доли за ~29 дней (информационный сигнал)
11. **Калибровка** = `signal_outcomes` логирует предсказания (по каждой комбинации quality/enchant) и раз в сутки сверяет их с фактическими продажами (`evaluate_signal_outcomes`) — для будущей подстройки констант

## 17. Тарифы и доступ (Phase 0 роадмапа подписок)

Источник истины — `backend/app/core/tiers.py`. 5 уровней доступа на пользователя:

| Тариф | Карточек watchlist | Telegram-уведомления | Окна статистики | Аукцион |
|---|---|---|---|---|
| `base` (дефолт после approve) | 6 | нет | 24ч | нет |
| `advanced` | 10 | да | 24ч+48ч | нет |
| `advanced_plus` | 20 | да | 24ч+48ч+7д | да |
| `advanced_max` | 25 | да | 24ч+48ч+7д+30д | да |
| `is_admin=True` | без лимита | да | все окна | да |

**Важное разделение:** `telegram_notifications` гейтит только проактивную отправку уведомлений (§15). Привязка самого Telegram-аккаунта (`/telegram/link-code`) — НЕ тарифная фича, доступна всем уровням одинаково (задумана как универсальный канал восстановления доступа; обязательная привязка при регистрации — отдельная, пока ОТЛОЖЕННАЯ пользователем фаза, см. `docs/NOTES.md`).

**Точки гейтинга:**
- Лимит watchlist — `POST /watchlist/` (403 при превышении).
- Доступ к аукциону — оба эндпоинта `GET /lots/...` (403 без доступа).
- Окна статистики — `GET /monitoring/item/{id}` маскирует поля ответа (не SQL-запрос) для неразрешённых окон; `GET /monitoring/sales-chart/{id}` и `GET /monitoring/history/{id}` возвращают пустой результат при запросе `hours`, превышающем максимум разрешённого окна (`max_stats_hours()`) — это были изначально незащищённые эндпоинты, дыра закрыта отдельным фиксом после первого прохода реализации.
- Telegram-уведомления — `telegram_bot/bot.py::notify_profitable_lots`.

**Авто-понижение при истечении `tier_expires_at`:**
- Лениво: при каждом запросе пользователя (`apply_tier_expiry` в `get_current_user`) — если тариф платный и просрочен, тут же понижается до `base`.
- Ежесуточный sweep (`sweep_expired_tiers`, §11) — гарантирует, что админка не показывает устаревший тариф у давно неактивных пользователей.
- При понижении лишние карточки `user_watchlist` сверх нового лимита **автоматически деактивируются** (`is_active=False`), оставляя активными самые старые по `created_at` — это сознательное решение (не блокировка доступа целиком, но и не «бесплатно сохранить повышенный лимит навсегда»).

**Авто-подтверждение регистрации** (`registration_settings`, синглтон): если включено админом — `register()` сразу выдаёт `is_approved=True` и тариф/срок по настройкам, минуя ручной approve. Выключено по умолчанию (поведение не меняется без явного действия админа).
