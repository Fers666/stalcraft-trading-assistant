 # Архитектура сбора данных

## Проблема наивного подхода

Если хранить сбор данных **per-user**, при росте числа пользователей возникает катастрофическое дублирование:

```
100 пользователей × 50 товаров в watchlist = 5000 API запросов каждые 5 минут
5000 × 2 токена = 10 000 токенов/5мин → лимит 100 токенов/мин → НЕВОЗМОЖНО
```

Также нет смысла собирать данные по одному `(item_id, region)` 100 раз —
рыночные данные одинаковы для всех пользователей.

---

## Решение: два слоя сбора

### Слой 1 — Watchlist (высокая частота, подробные данные)

**Кто:** объединение уникальных `(item_id, region)` пар из watchlist **всех** пользователей.
**Как часто:** каждые 5 минут.
**Что собирает:** полный снэпшот лотов — raw_lots, liquid/expiring split, snapshot-history matching.
**Где хранит:** `collected_data` с `user_id = NULL` (глобальная запись).
**Эффект:** 100 пользователей следят за одним товаром → **1 API запрос**, не 100.

```
Watchlist всех пользователей:
  user1: [m02wr/RU, 04yr/RU]
  user2: [m02wr/RU, y1q9/EU]
  user3: [m02wr/RU]

Дедупликация → уникальные пары:
  m02wr/RU, 04yr/RU, y1q9/EU

Результат: 3 API запроса вместо 7
```

### Слой 2 — Global Feed (низкая частота, лёгкие данные)

**Кто:** все предметы из `master_items` **кроме** тех что уже в watchlist хоть у кого-то.
**Как часто:** скользящий цикл ~24 часа (каждый час — один батч из ~93 предметов).
**Что собирает:** только ключевые метрики (цена, количество лотов, объём).
**Где хранит:** `global_item_scan` — лёгкая таблица без raw JSON.
**Цель:** лента "интересных" предметов для discovery — найти выгодный товар до того как его кто-то добавил в watchlist.

```
master_items: 2236 предметов
  └── В watchlist (у кого-либо): исключаем → уже собирается каждые 5 мин
  └── Остальные: ~2186 шт → глобальный скан раз в сутки
```

**Почему скользящий, а не ночной:**
Ночные данные отражают тихий рынок, а не реальную торговую активность.
Скользящий цикл захватывает прайм-тайм естественно — каждый час идёт батч,
данные актуальны в любое время суток включая вечерний пик.

## Персонализация

Сбор данных глобальный, персонализация — на уровне запроса:

| Что | Уровень |
|-----|---------|
| Цены, лоты, история | Глобальный (1 запись на пару) |
| Мин. маржа (10%) | Личный (user_settings) |
| Уведомления | Личный (user_settings) |
| Tracked batch sizes | Личный (user_watchlist) |
| Ручной refresh | Личный (user_id в collected_data) |
| Статистика по quality/enchant | На лету (фильтрация sales_history при запросе) |

### Фильтрация по quality/enchant (карточка Избранного)

`market_statistics` хранит глобальные агрегаты без разбивки по качеству/заточке.
Когда watchlist-запись имеет `quality_filter` или `enchant_filter`, три блока карточки получают данные по-разному:

| Блок | Источник | Как фильтруется |
|------|---------|-----------------|
| Выгодные лоты | `GET /lots/{id}` + фронт | `profitableLots` сравнивает `quality_name`/`enchant_level` |
| Варианты продажи | `GET /monitoring/item/{id}?quality_filter=…` | На лету: `median_price_7d`, `sales_volume_7d`, `sell_options` из отфильтрованных `sales_history`; `best_sell_hour`/`buy_hours` и проч. берутся из глобальной `market_statistics` |
| История продаж | `GET /monitoring/sales-chart/{id}?quality_filter=…` | SQL-фильтр по `additional_info->>'qlt'` / `upgrade_bonus` |

`sell_options` в режиме с фильтром считаются с `confidence=low` (только объём, без lot_start пар), поскольку выборка по одному качеству/заточке обычно мала для статистической уверенности.

## Таблица `global_item_scan` (новая)

Лёгкие метрики для предметов вне watchlist.

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | integer PK | |
| `item_id` | varchar(50) FK → master_items | |
| `region` | varchar(10) | |
| `scanned_at` | timestamptz | Время скана |
| `lot_count` | integer | Число активных лотов |
| `liquid_lot_count` | integer | Лотов с остатком ≥ 2ч |
| `best_price` | bigint | Лучшая цена выкупа |
| `avg_price` | numeric(12,2) | Средняя цена |
| `total_volume` | integer | Суммарное кол-во единиц |
| `prev_best_price` | bigint | Лучшая цена прошлого скана |
| `price_change_pct` | numeric(5,2) | Изменение цены, % |
| `tradability_score` | numeric(8,2) | Скор торгуемости (см. ниже) |

**UNIQUE:** `(item_id, region)` — одна актуальная запись, обновляется при каждом скане.

### Скор торгуемости (tradability_score)

```
tradability_score = lot_count × liquid_lot_count × (1 / (1 + price_spread_pct))
```

Где `price_spread_pct` — насколько разброс цен велик (высокий = нестабильный рынок).
Чем больше ликвидных лотов и меньше разброс — тем выше скор.

## Изменения в существующих таблицах

### `collected_data`

`user_id` становится **nullable**:
- `user_id = NULL` → глобальный снэпшот (из watchlist коллектора)
- `user_id = <id>` → ручной refresh конкретного пользователя

### `market_statistics`

`user_id` становится **nullable**:
- `user_id = NULL` → глобальная статистика (одна на пару item/region)
- Пользователи читают глобальную статистику, применяют личные фильтры при отображении

### `sales_history`

`user_id` **NOT NULL** — отличие от двух таблиц выше.  
Запись принадлежит пользователю, чей watchlist вызвал сбор. Однако данные фактически глобальные:

- Дедупликация по `(item_id, region, sale_time)` происходит **без фильтра по user_id** → каждая реальная сделка хранится ровно один раз, независимо от числа пользователей, следящих за товаром.
- API `/monitoring/sales-chart` читает данные без фильтра по `user_id` — история аукциона публична.
- Следствие: первый коллектор, который запустится для данного `(item_id, region)`, "присваивает" себе запись. Это технический артефакт, не несущий смысловой нагрузки.

## Celery расписание (обновлённое)

| Задача | Расписание | Описание |
|--------|-----------|----------|
| `collect_all_active_lots` | каждые 5 мин | Уникальные пары из всех watchlist |
| `collect_all_history` | раз в час (мин. 0) | История для watchlist предметов |
| `calculate_all_market_stats` | раз в час (мин. 5) | Пересчёт market_statistics |
| `run_global_feed_batch` | раз в час (мин. 30) | Батч из ~93 предметов вне watchlist |
| `delete_old_data` | ежедневно 03:00 | Удаление данных старше 120 дней |

### Разовые задачи (вызываются из API)

| Задача | Триггер | Описание |
|--------|---------|----------|
| `collect_single_item` | Добавление в watchlist / ручной refresh | Снэпшот лотов для одного товара |
| `collect_history_single` | Добавление в watchlist | История продаж для одного товара |
| `calculate_stats_single` | Добавление в watchlist | Пересчёт market_statistics для одного товара |

**Цепочка при добавлении товара в watchlist:**

```
POST /watchlist/ → Celery chain:
  collect_single_item   (лоты)     ~5–10 сек
       ↓
  collect_history_single (история) ~10–20 сек
       ↓
  calculate_stats_single (статистика) ~2 сек
```

Карточка товара заполняется за ~30–60 секунд, а не ожидает планировщика (до 55 минут).

### Автообновление фронтенда

`MonitoringPage` поллит `loadAll()` каждые 5 минут — соответствует интервалу `collect_all_active_lots`.
Данные появляются на экране без ручного обновления страницы.
