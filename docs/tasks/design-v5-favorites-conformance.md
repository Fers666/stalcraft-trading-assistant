# ТЗ: Design v5 «Избранное» — приведение к эталону-прототипу (conformance)

## Контекст

Фаза 3 (страница «Избранное») внедрена, но на ревью пользователь нашёл отклонения
от эталонного прототипа `design/v5/app/favorites.html`. Требование: **точное
соответствие раскладке, расположению блоков, анимациям выбора и рендеру иконок**.
«Переносим дизайн-систему» не означает свободу менять компоновку карточки.

Это ТЗ — покомпонентная сверка прототипа с реализацией и предписание правок для
`frontend-dev`. Бизнес-логика расчётов (§8 общей спеки) НЕ трогается — меняем только
раскладку, разметку и стили.

**Приоритет источников:** при конфликте общей спеки `design-v5-implementation.md` §5.1
и прототипа `favorites.html` — **побеждает прототип** (явно подтверждено пользователем).

## Пункт 1 — Раскладка карточки (ГЛАВНОЕ). Вердикт: ОТКЛОНЕНИЕ

**Прототип:** 4 ячейки `.cell` в одной сетке `.grid-2` внутри панели `.panel#card`:
ряд 1 `[Выгодные лоты | Динамика цен(графики+табы 24Ч/48Ч/7Д/30Д)]`,
ряд 2 `[Варианты продажи | Пачки·распределение]`. Щели между ячейками — 1px фона `--line`
(достигается `gap:1px` + `background:var(--line)` на контейнере и непрозрачным `bg1` на ячейках).

**Реализация:** `MonitoringPage:156-175` рендерит `<LotStatCard/>` и `<SalesHistoryCharts/>`
**вертикальным стеком** (`flexDirection:'column', gap:3`). Внутри `LotStatCard:462` сетка —
только 2 ячейки `[Выгодные лоты | (Варианты продажи + Пачки, стек в одной ячейке)]`.
«Динамика цен» вынесена отдельным блоком под карточкой — этого в эталоне нет.

**Предписание (точно):**
1. В `LotStatCard` сделать сетку из **4 ячеек**. Контейнер (заменить `LotStatCard:462`):
   ```
   sx={{ display:'grid',
         gridTemplateColumns:{ xs:'1fr', ... }  // desktop: 'minmax(0,1.15fr) minmax(0,1fr)'
         gap:'1px', background: tokens.border, borderTop:`1px solid ${tokens.border}` }}
   ```
   Точки останова — как в base.css:148-149: одна колонка при ≤1380px.
2. Порядок ячеек в DOM (auto-flow заполнит 2×2): **①Выгодные лоты → ②Динамика цен → ③Варианты продажи → ④Пачки**.
3. Ячейка ②Динамика: перенести `SalesHistoryCharts` внутрь `LotStatCard` как содержимое ячейки:
   `<Box sx={{ background: tokens.bg1, p:'12px 16px 16px', minWidth:0 }}><SalesHistoryCharts itemId={itemId} region={region} qualityFilter={qualityFilter} enchantFilter={enchantFilter} median={stats?.median_price_7d ?? undefined} /></Box>`.
   `SalesHistoryCharts` уже рендерит `.sec-h` (h2 «Динамика цен» + табы) — совпадает с эталоном.
   **Прокинуть `median`** (сейчас `MonitoringPage` его не передаёт → линия-ориентир не рисуется).
4. Ячейки ③ и ④: разъединить нынешнюю комбо-ячейку `LotStatCard:573-690` на две отдельные `.cell`-Box.
5. В `MonitoringPage` удалить `import`/рендер `SalesHistoryCharts` и обёртку-стек; центральная колонка = `<LotStatCard fullWidth .../>`.

**Trade-off / архитектура:** переносим `SalesHistoryCharts` **внутрь** `LotStatCard`, а не
композируем в `MonitoringPage`, потому что эталон требует единой панели `.panel` с общей
сеткой `.grid-2` (одна рамка, общие 1px-щели). Композиция снаружи даёт две отдельные рамки/зазор
и ломает компартмент-сетку. `SalesHistoryCharts` остаётся отдельным компонентом (переиспользуемым),
просто монтируется как children ячейки — минимальная хирургия, дублирования логики нет.

**Заметки (не блокеры, отметить frontend-dev):**
- Сейчас grid рендерится только при наличии `stats` (`LotStatCard:461`). После переноса графика
  в ячейку — график окажется скрыт, когда `stats == null`. В эталоне ячейки существуют всегда и
  каждая показывает своё пустое состояние. **Рекомендация:** рендерить `.grid-2` при наличии
  выбранного предмета (не гейтить на `stats`); ячейки Варианты/Пачки/Выгодные сами показывают
  «нет данных». Как минимум — ячейка графика должна быть независима от `stats`.
- Toggle «Сегодня/Неделя» (`LotStatCard:447-455`) и toggle «Сейчас/Неделя» в шапке «Выгодных
  лотов» (`LotStatCard:478-483`) — **в прототипе отсутствуют** (эталонная `.sec-h` = h2 + `.cnt` +
  `.hint`«клик по строке — пересчёт вариантов продажи»). Это продуктовое обогащение под реальные
  данные. НЕ требование данного ревью, но флажок: они добавляют элементы, которых нет в эталоне.
  Оставить на усмотрение — при строгом соответствии заменить hint-текстом. Согласовать с пользователем.

## Пункт 2 — Иконки. Вердикт: URL-формула ВЕРНА, причина — данные/сеть (см. гипотезы)

**Сверка пути:**
- Демо-данные (`data.js`): `iconBase = ".../stalcraft-database/main/ru"`, поле `icon` = **полный**
  URL, напр. Гелий(kqr0): `.../main/ru/icons/artefact/electrophysical/kqr0.png`.
- React: `ICON_BASE = ".../main/ru"` (i18n.ts:134), `iconUrl(p) = ICON_BASE + p` (i18n.ts:137-140).
- Backend: `master_items.icon_path` хранит путь вида `/icons/<категория>/<id>.png`
  (models.py:75, github_parser.py:65, миграция 0008: «Полный URL = .../main/ru{icon_path}»).

→ `ICON_BASE + icon_path` = `.../main/ru/icons/<cat>/<id>.png` — **точно совпадает с демо-URL.
Формула построения URL корректна.** `ItemIcon` тоже корректен (img → фолбэк-буква на `onError`).

**Значит, «картинки не отображаются» — это НЕ баг склейки URL. Гипотезы (по убыванию вероятности):**

- **H1 (наиболее вероятно): `icon_path == NULL` в БД для избранных предметов.** Тогда
  `iconUrl` → `null`, `ItemIcon` показывает **фолбэк-букву на цвете качества** (не «битую»
  картинку). Совпадает с симптомом «нет картинок». Причина: backfill каталога/иконок неполон
  для части предметов (в MEMORY — незакрытые backfill'ы), либо предмет добавлен до синка
  `github_parser`. Проверка: в ответе `/watchlist` (или в сторе) поле `icon_path` = null.
- **H2: `icon_path` в БД без ведущего `/`** (напр. `icons/...`) → URL `.../main/ruicons/...` → 404.
  `github_parser` кладёт `entry.get("icon")` с ведущим слэшем, но у части записей формат мог
  отличаться. Проверка: посмотреть фактическое значение в API-ответе.
- **H3: сеть/блокировка `raw.githubusercontent.com` в браузере пользователя** (adblock, DNS,
  корпоративный фильтр). Картинки грузятся клиентом, не сервером. Проверка: в Network — запрос
  есть, но `failed`/`blocked`/`(canceled)`.
- **H4 (низкая): CORS/404 конкретного файла** — если id/категория в `icon_path` не соответствуют
  реальному файлу в репозитории EXBO. Проверка: запрос есть, статус `404`.

**Что должен проверить qa-tester (Network tab):**
1. Открыть «Избранное», DevTools → Network → фильтр `githubusercontent`.
2. **Ожидаемый рабочий URL** (пример-эталон, гарантированно существует в репо):
   `https://raw.githubusercontent.com/EXBO-Studio/stalcraft-database/main/ru/icons/artefact/electrophysical/kqr0.png`
   — открыть напрямую в браузере: если картинка грузится, сеть/репозиторий в порядке.
3. Диагностика по симптому:
   - **Запроса иконки нет вообще** → `icon_path` = null (H1). Проверить JSON ответа `/watchlist`
     и `/monitoring/item/...` на поле `icon_path`. Эскалация — на `backend-dev` (backfill icon_path).
   - **Запрос есть, статус 404** → путь битый (H2/H4): сравнить фактический `icon_path` с реальным
     путём файла в репозитории.
   - **Запрос есть, `blocked`/`failed`** → сеть/адблок пользователя (H3), не код.

**Правок в коде иконок НЕ требуется**, пока qa-tester не подтвердит гипотезу. Если H1 — задача
уходит `backend-dev` (backfill `icon_path`), а не `frontend-dev`.

## Пункт 3 — Анимации/стили выбора. Вердикт: САЙДБАР — ОТКЛОНЕНИЕ; таблица — ОК

### 3a. Сайдбар `.fav` — избранные строки перекрашены зелёным (ОТКЛОНЕНИЕ)

**Прототип** (`.fav`, base.css? нет — favorites.html:69-95):
- `transition: background-color, border-color` (только эти два, `--fast`).
- `:hover → background:var(--s2)`; `:active → background:var(--s3)`.
- `.fav.on → background:var(--gold-dim); border-left-color:var(--gold-hi)`; `.fav.on .f-name → color:var(--gold-2)`.
- **Выгодность НЕ красит строку.** Признак выгодности — только правый блок `.f-right`:
  `.f-med` (медиана 7д, `fmtCompact`, серый) + зелёный бейдж `.f-sig` «+N» (кол-во выгодных
  лотов), показывается лишь при N>0. Плюс `.f-plus` «+заточка» рядом с именем.

**Реализация** (`MonitoringPage:270-320`):
- `borderLeftColor: isProfitable ? tokens.success` — **зелёный бордер** у выгодных ❌
- `background: isProfitable && !isSelected ? tokens.successDim` — **зелёный фон** ❌
- `'&:hover': isProfitable ? successDim` — зелёный hover (эталон: всегда `s2`) ❌
- Зелёный квадрат-дот 6px перед именем (`:293-295`) — **в эталоне отсутствует** ❌
- `color: isProfitable ? tokens.success` у имени (`:299`) — эталон: имя `--text`, gold-2 только при `.on` ❌
- **Отсутствует** правый блок `.f-right`: нет `.f-med` (медиана) и нет `.f-sig` «+N» ❌

**Предписание:**
1. Убрать всю зелёную индикацию строки: `background`, `borderLeftColor`, `&:hover` для
   `isProfitable`, зелёный дот, зелёный цвет имени.
2. Оставить состояния строго по эталону:
   - hover → `tokens.bg2`; active → `tokens.bg3`.
   - selected (`.on`/`Mui-selected`) → `background: tokens.goldDim; borderLeftColor: tokens.goldHighlight`;
     имя → `tokens.goldAccent`.
   - `borderLeft:'2px solid transparent'` по умолчанию; `transition:'background-color, border-color'`.
3. Добавить правый блок `.f-right` (flex-none, text-align:right): строка `.f-med` — медиана 7д
   `fmtCompact` (`tokens.text1`, mono) + при выгодности зелёный бейдж `.f-sig` «+N»
   (`color:success; background:successDim; border:1px successLine; padding:0 5px; radius`).
4. `.f-plus` «+заточка» рядом с именем — уже есть (`:302-306`), оставить.

**Open question (данные для сайдбара):**
- `.f-med` требует медиану 7д на предмет. `.f-sig` «+N» требует **счётчик** выгодных лотов, а
  стор отдаёт лишь `profitableItemIds` (членство, без числа). Нужно определить: (а) surface-ить
  медиану/счётчик в `FeedWatchlistEntry`/сторе, либо (б) если счётчик недоступен — показывать
  бейдж-маркер без числа. **Рекомендация:** уточнить у `backend-dev`/в сторе, есть ли счётчик
  выгодных лотов на предмет; если нет — минимальный вариант (бейдж-маркер) как временное решение,
  с задачей на surface счётчика. Это НЕ меняет бизнес-логику расчётов.

### 3b. Строка таблицы лотов `tr[aria-selected]` — ОК (соответствует)

**Прототип** (base.css:185-190): `tr` transition `background-color --fast`; hover `s2`; active `s3`;
`tr[aria-selected="true"] → background:gold-dim; box-shadow:inset 2px 0 0 gold-hi; td color gold-2;
td.pos color green (профит остаётся зелёным)`.

**Реализация** (theme.ts:353-367 `MuiTableRow`): transition `background-color FAST`; `:hover td → BG2`;
`:active td → BG3`; `.Mui-selected td → background GOLD_DIM, color G3(gold-2)`;
`.Mui-selected → box-shadow inset 2px 0 0 G4(gold-hi)`. **Совпадает.** `LotStatCard:522-529`
корректно ставит `selected`+`aria-selected`.

**Единственная мелочь:** эталон сохраняет **зелёный** цвет колонки профита в выбранной строке
(`td.pos`), а theme `.Mui-selected td { color:G3 }` может перекрыть inline-цвет профита
(`LotStatCard:549`) → станет золотым. Проверить в браузере; при необходимости добавить в theme
`'&.Mui-selected td.pos': { color: SUCCESS }` (или повысить специфичность inline). Низкий приоритет.

**Прототип** (favorites.html:117-124) — 6 колонок:
`Цена выкупа | Кол-во | Цена/шт | Заточка | Осталось | Профит vs медиана` (одна колонка профита =
`медиана 7д − цена/шт`). Данные демо не содержат `sell_options` пер-лот, поэтому профит один.

**Реализация** (`LotStatCard:506-562`) — колонки:
`Цена/шт (+ «N шт · выкуп» подстрокой) | [Качество, опц.] | Быстро | Нормально | Выгодно`
(три колонки профита по стратегиям продажи, каждая с «итого»).

**Анализ.** Это **осознанное продуктовое отличие**, а не случайное отклонение: реальные данные
(`stats.sell_options` / `signals`) дают три сценария продажи (fast/normal/premium), которых в demo
нет. Три колонки профита = ценность продукта (§8: расчёты прибыли — не трогать). Однако относительно
эталона реализация **теряет** информативные колонки `Кол-во`, `Заточка`, `Осталось (время жизни лота)`.

**Рекомендация (компромисс, без изменения бизнес-логики):**
- **Сохранить** три колонки профита по стратегиям — это продуктовая ценность, отказ был бы регрессом.
- **По возможности вернуть эталонные колонки** там, где данные есть в `LotItem`/`SignalLot`:
  - `Кол-во` (`amount`) — есть, сейчас спрятано в подстроку. Можно вынести отдельной колонкой (как в эталоне).
  - `Заточка` (`enchant_level`/`ptn`) — есть; эталонная колонка «Заточка» (`+N`/`—`) вместо/вдобавок к «Качество».
  - `Осталось` (`hours_remaining`/`is_expiring`) — есть в `LotItem`, но **отсутствует в `signals.lots`**
    (`SignalLot` не содержит времени истечения). Поэтому колонку «Осталось» можно показать только на
    ветке fallback-лотов, не на ветке сигналов → неконсистентно. **Вывод:** «Осталось» — опционально,
    только если появится в signals; иначе не добавлять, чтобы не плодить пустую колонку.
- **Не считать это блокером conformance.** Отметить как согласование с пользователем: «оставляем 3
  колонки профита (продуктовое преимущество над demo), при этом возвращаем `Кол-во` и `Заточка`
  отдельными колонками для близости к эталону; `Осталось` — только при наличии данных в сигналах».
- Заголовки сортируемых колонок — через `ui/SortHeader` с `aria-sort` (уже используется).

**Требует подтверждения пользователя:** финальный набор колонок (3 стратегии профита vs. одна
колонка «Профит vs медиана» как в эталоне). Рекомендация — гибрид выше; решение за пользователем.
