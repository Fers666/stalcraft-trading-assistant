# ТЗ: market_stats — размазывание пересчёта по часу + дифф-пропуск пар без новых продаж

## Контекст

`calculate_all_market_stats` (backend/app/tasks/analyzers.py:44) — главный источник
часовых CPU-пиков 60–80% на 2-vCPU проде. Прод-замеры 2026-07-07: 193–236 секунд
каждый час, рост +20% за 13ч, 85 уникальных пар `(item_id, region)` → ~2.7с/пара.
Задача однопоточная, CPU-bound: на каждую пару из БД тянутся ВСЕ продажи за 30 дней
(ORM) и ВСЕ снэпшоты `collected_data` за 30 дней (десятки тысяч строк на пару,
см. комментарий у запроса `snapshots_30d` в
backend/app/services/analytics/market_stats.py:212).

Сейчас задача запускается цепочкой `calculate_all_market_stats.delay()` в конце
`collect_all_history` (backend/app/tasks/collectors.py:221-222; раз в час,
crontab minute=0). Цепочка появилась в коммите `9f8086e` намеренно — чтобы
history и stats не грузили оба ядра одновременно. Микрооптимизации Python-кода
уже сделаны (`_fast_mean`, однопроходные бакеты, коммит `9a8442d`) — недостаточно.
Предыстория: docs/tasks/cpu-spikes-recurring-2026-07-06.md (там это пункт 4 плана).

Согласованное с пользователем направление: (а) размазать пересчёт порциями по
часу, (б) пропускать пары без новых продаж, с принудительным полным пересчётом
раз в сутки ночью.

**Ожидаемый эффект:** вместо монолитных 4 минут CPU на одном ядре — 10 коротких
порций по ~10–25с в час, причём большинство пар в тихие часы пропускаются по
диффу вовсе (перепроверяются только пары с новыми продажами).

## Решение (архитектурные решения и обоснование)

### 1. Деление на порции: hash-слоты, без состояния в Redis

Каждая пара детерминированно закреплена за одним из `NUM_SLOTS = 10` слотов:

```python
slot_of_pair = zlib.crc32(f"{item_id}|{region}".encode()) % NUM_SLOTS
```

Слот для текущего запуска вычисляется из wall-clock (минута по МСК,
`timezone(timedelta(hours=3))` — тот же паттерн, что в market_stats.py):

```python
slot = ((minute_msk - 12) // 5) % NUM_SLOTS   # расписание :12, :17, ..., :57
```

**Почему hash, а не курсор в Redis и не срез отсортированного списка:**
- Stateless: нет Redis-ключа, который может потеряться/разъехаться (свежий
  инцидент с утечкой Redis-соединений — лишнее состояние в Redis не добавляем).
- Устойчивость к изменению watchlist между порциями: добавление/удаление пары
  НЕ сдвигает слоты остальных (срез по индексу в отсортированном списке при
  вставке сдвигает всё → пары пропускаются или считаются дважды за круг).
  Новая пара сразу попадает в свой постоянный слот.
- Равномерность: crc32 на 85 парах даёт ~8–9 пар на слот; при 2.7с/пара —
  порция ~23с. Это приемлемо для общей очереди `collector`
  (worker_concurrency=2): `collect_all_active_lots` задержится максимум на
  ~23с вместо нынешних 4 минут.

Компромисс: распределение не идеально ровное (возможен слот с 12 парами и слот
с 5) — принимаем, разброс длительности порции 15–35с не критичен.

### 2. Конфликт с collect_all_history: сдвиг фаз вместо цепочки

- Цепочка `calculate_all_market_stats.delay()` из `collect_all_history`
  **удаляется полностью**. Порции живут на независимом beat-расписании.
- Расписание порций: `crontab(minute="12-59/5")` → 10 запусков в час
  (:12, :17, ..., :57). Окно :00–:11 свободно — его целиком занимает
  `collect_all_history` (по прод-графикам плато истории ~10 минут, :01–:11).
  Мотивация `9f8086e` (не пересекаться с history) сохранена сдвигом фаз.
- Если история затянется дольше :12 — порция просто подождёт в FIFO-очереди
  и выполнится следом; при вычислении слота от wall-clock задержавшаяся порция
  может обработать соседний слот, тогда часть пар пропустит один час — это
  ограничено сверху ночным принудительным пересчётом (staleness ≤ 24ч+1ч).
  Осознанно НЕ добавляем компенсацию (Redis-счётчик слотов) — сложность не
  окупается.

### 3. Дифф-пропуск: один SQL-запрос на порцию, по `collected_at`

Перед циклом порции — ОДИН запрос, возвращающий множество «грязных» пар:

```sql
SELECT DISTINCT sh.item_id, sh.region
FROM sales_history sh
JOIN market_statistics ms
  ON ms.item_id = sh.item_id AND ms.region = sh.region
WHERE sh.collected_at > ms.calculated_at
  AND sh.collected_at >= now() - interval '26 hours'
```

- **Критично: сравниваем `collected_at`, а НЕ `sale_time`.** `sale_time` —
  игровое время сделки, оно всегда отстаёт от момента вставки строки (история
  собирается раз в час): строка, вставленная в 12:05 с `sale_time` 11:40, при
  фильтре по `sale_time > calculated_at` (расчёт был в 11:50) была бы потеряна.
  `collected_at` — момент попадания строки в БД, семантика диффа именно такая.
- Нижняя граница `now() - 26h` корректна: ночной полный пересчёт обновляет
  `calculated_at` всех пар раз в сутки, значит строка старше 26ч не может быть
  новее чьего-либо `calculated_at`. Граница нужна планировщику, чтобы работать
  по новому индексу, а не сканом 120-дневной таблицы.
- **Новый индекс** btree на `sales_history.collected_at` (Alembic-миграция,
  следующая ревизия после текущей head; текущие: 0031_emission_events).
  Существующие индексы (`ix_sales_item_time`, `ix_sales_cleanup`) этот запрос
  не покрывают.
- Пара из слота обрабатывается, если она в «грязном» множестве **ИЛИ** у неё
  вообще нет строки `MarketStatistics` (join её не вернёт — множество пар,
  имеющих MS-строку, получаем тем же или отдельным лёгким запросом). Новая
  пара watchlist считается сразу в свой ближайший слот, не ждёт ночи.
  (Основной путь для новых пар — существующая цепочка
  `POST /watchlist → ... → calculate_stats_single`, backend/app/api/v1/endpoints/watchlist.py:143-147
  — не трогаем; проверка в слоте — защитная сетка на случай сбоя цепочки.)

**Снэпшоты и best_buy_hour — осознанный компромисс.** Поля
`best_buy_hour/best_buy_day/buy_hours_by_day` зависят от снэпшотов
`collected_data`, которые появляются постоянно (каждые ~20с–5мин). Если считать
новые снэпшоты признаком «грязности» — дифф выродится: все пары всегда грязные.
Решение: снэпшоты НЕ триггерят пересчёт. Обоснование: это агрегаты по часам
суток за 30 дней — вклад одних суток новых снэпшотов в среднее по часу
пренебрежим, суточной свежести (ночной пересчёт) достаточно. Для пар С новыми
продажами best_buy обновляется как раньше — при их пересчёте.

Аналогично устаревают максимум на сутки у неторгуемых пар окна 24ч/48ч/7д
(старые продажи «выпадают» из окна без новых) — именно поэтому полный пересчёт
раз в сутки обязателен, чаще не требуется (подтверждено пользователем).

Альтернатива «`collect_all_history` пишет dirty-пары в Redis-set» отвергнута:
связывает задачи, добавляет теряемое состояние в Redis, а SQL-запрос по
индексу и так стоит миллисекунды раз в 5 минут.

### 4. Ночной принудительный пересчёт: force-час внутри тех же слотов

Отдельную beat-запись «полный пересчёт в 4:00» НЕ добавляем — в 4:00 стартует
ежечасный `collect_all_history` (crontab minute=0 — каждый час, включая 4:00),
монолитный 4-минутный пересчёт рядом с ним вернул бы ночной пик.

Вместо этого: `FULL_RECALC_HOUR_MSK = 4` — порции, выполняющиеся в час 04 по
МСК (слоты 04:12–04:57), игнорируют дифф-фильтр и пересчитывают ВСЕ пары своего
слота (`force=True` в логах). За час 04:12–04:57 полный круг из 85 пар проходит
теми же порциями по ~23с — ночью пик остаётся плоским. Соседние ночные задачи
не мешают: cleanup 3:00, sweep_tiers 3:30, evaluate_signal_outcomes 4:30
(пересечение с одной 20-секундной порцией — пренебрежимо).

### 5. Судьба calculate_all_market_stats

Задача остаётся в коде как ручной инструмент полного пересчёта (вызов через
`docker exec ... celery call`), из beat её нет (уже нет), из цепочки — убираем.
Retry-политика (`max_retries=3, countdown=120`) переносится на новую порционную
задачу.

## Детали реализации по файлам

### backend/app/tasks/analyzers.py

1. Константы модуля: `NUM_SLOTS = 10`, `SLOT_FIRST_MINUTE = 12`,
   `SLOT_STEP_MINUTES = 5`, `FULL_RECALC_HOUR_MSK = 4`.
2. Новая задача:

```python
@celery_app.task(name="app.tasks.analyzers.calculate_market_stats_batch",
                 bind=True, max_retries=3)
def calculate_market_stats_batch(self):
    """Порционный пересчёт market_statistics: 1/NUM_SLOTS активных пар,
    полный круг за час. В час FULL_RECALC_HOUR_MSK — force-пересчёт без диффа."""
```

   Внутри `_run()` (паттерн `run_async` как у существующих задач):
   - `now_msk = datetime.now(timezone(timedelta(hours=3)))`;
     `slot = ((now_msk.minute - SLOT_FIRST_MINUTE) // SLOT_STEP_MINUTES) % NUM_SLOTS`;
     `force = now_msk.hour == FULL_RECALC_HOUR_MSK`.
   - Выборка активного watchlist и дедупликация пар — как в текущей
     `calculate_all_market_stats` (analyzers.py:54-60).
   - `slot_pairs = [p for p in unique_pairs if zlib.crc32(...) % NUM_SLOTS == slot]`.
   - Если не `force`: одним запросом получить `dirty_pairs` (SQL из раздела 3;
     через SQLAlchemy `select(...).join(...)` или `text()` — на выбор
     backend-dev, важна семантика) и множество пар, имеющих строку
     `MarketStatistics`; `to_process = [p for p in slot_pairs
     if p in dirty_pairs or p not in pairs_with_stats]`.
     Если `force`: `to_process = slot_pairs`.
   - Цикл по `to_process` — тело как сейчас (analyzers.py:63-80): вызов
     `calculate_market_stats`, per-pair try/except с `db.rollback()`.
   - Логи (см. «Наблюдаемость»).
3. `calculate_all_market_stats` — оставить без изменений поведения (ручной
   полный пересчёт), обновить докстринг («ручной инструмент, из расписания
   и цепочки убрана — см. calculate_market_stats_batch»).
4. `calculate_market_stats` в market_stats.py — **НЕ трогать** (ни формулы,
   ни запросы).

## Критерии успеха

- [ ] Grafana, окно :00–:10 каждого часа: нет изолированного пика >60% на ядро
      от статистики после плато истории (характерный «второй скачок» через
      20–40 минут после часа исчезает).
- [ ] Логи worker: 10 записей `batch: slot=` в час, каждая с длительностью;
      длительность порции ≤ ~35с; в тихие часы `calculated` заметно меньше
      `pairs_in_slot` (дифф работает).
- [ ] В час 04 МСК все порции идут с `force=True` и суммарно за час
      пересчитаны все активные пары.
- [ ] SQL-проверка на проде спустя >26ч после деплоя: `SELECT count(*) FROM
      market_statistics ms JOIN (SELECT DISTINCT item_id, region FROM
      user_watchlist WHERE is_active) w USING (item_id, region) WHERE
      ms.calculated_at < now() - interval '25 hours'` → 0 строк.
- [ ] Новая пара, добавленная в watchlist, получает MarketStatistics в течение
      часа даже при сбое цепочки `calculate_stats_single` (защитная ветка
      «нет MS-строки» в слоте).
- [ ] `EXPLAIN` дифф-запроса использует `ix_sales_collected_at`.
- [ ] Значения полей MarketStatistics при пересчёте идентичны прежним
      (формулы не менялись — регресс-сравнение одной пары до/после).
- [ ] Цепочка `POST /watchlist → calculate_stats_single` работает как раньше.
