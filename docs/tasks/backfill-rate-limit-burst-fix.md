# ТЗ: Фикс burst 429 в разовом backfill-скрипте qlt/ptn

## Контекст

При запуске на проде `backend/app/scripts/backfill_sales_qlt.py --days 30 --yes`
(разовый бэкфилл `qlt`/`ptn` в `sales_history`, см. `docs/NOTES.md`, коммит
`64ecb3c`) скрипт почти сразу начал стабильно падать в `429 Too Many Requests`
почти на каждом запросе и не продвигался — только спал 60с и ронял всю текущую
пару (item_id, region) без реального прогресса. Пользователь остановил скрипт
вручную (Ctrl+C).

Причина — не превышение документированного лимита Stalcraft API (400 запросов/мин),
а комбинация трёх независимых проблем, каждая по отдельности безобидна, но вместе
дающая взрывной отказ:

1. **Token bucket ограничивает только средний расход за минуту, не пиковую
   скорость.** `TokenBucketRateLimiter._LUA_ACQUIRE` (`backend/app/core/rate_limiter.py:54-83`)
   атомарно проверяет накопленные токены и сразу отдаёт их все, если они есть в
   запасе — бакет не навязывает минимальный интервал между последовательными
   `acquire()`. Если в момент запуска backfill в бакете был резерв (например,
   между запросами обычного сбора), скрипт мог выгрести этот резерв практически
   мгновенно, а затем продолжать запрашивать токены быстрее, чем бакет успевает
   их восполнять (400/60 ≈ 6.67 токенов/сек, `HISTORY` стоит 2 токена/запрос →
   максимум ~3.3 запроса/сек устойчиво) — наблюдали ~5 запросов/сек.

2. **`_backfill_pair()` не делает пауз между постраничными запросами.**
   `backend/app/scripts/backfill_sales_qlt.py:141-196` — цикл `while True`,
   каждая итерация сразу зовёт `stalcraft_client.get_auction_history(...)` со
   следующим `offset`, без `asyncio.sleep` между итерациями. Единственное
   естественное ограничение — сетевая задержка ответа (round-trip), отсюда
   наблюдаемые ~5 запросов/сек. Это сильно отличается от профиля обычного
   почасового сбора (`_collect_history_for_item`, `backend/app/tasks/collectors.py:518-550`),
   который делает **один** запрос на пару (item_id, region) без пагинации — у
   backfill же одна "толстая" пара истории за 30 дней может требовать десятки
   страниц подряд.

3. **При 429 `client.py::_request()` не делает retry текущей страницы — рейзит
   исключение, которое гасит всю пару.** `backend/app/services/collector/client.py:45-48`:
   ```python
   if response.status_code == 429:
       logger.error("429 received despite token bucket — backing off 60s")
       await asyncio.sleep(60)
       raise RuntimeError("Rate limit exceeded (429)")
   ```
   Это исключение всплывает в `run()` (`backend/app/scripts/backfill_sales_qlt.py:230-234`):
   ```python
   try:
       requests_spent, updated = await _backfill_pair(db, item_id, region, cutoff)
   except Exception as e:
       logger.error(f"[{i}/{len(pairs)}] {item_id}/{region} failed: {e}")
       continue
   ```
   — ловится на уровне ВСЕЙ пары, не текущей страницы. Прогресс по offset внутри
   `_backfill_pair()` теряется (offset — локальная переменная функции, не
   сохраняется), при повторном запуске скрипта пара (если в ней остались строки
   без qlt) начнётся с `offset=0` заново. На практике после первого 429 почти
   каждая следующая страница той же пары снова ловит 429 (бакет уже исчерпан, а
   скрипт продолжает долбить на полной сетевой скорости сразу после `sleep(60)`),
   так что пара фактически никогда не продвигается дальше первых страниц.

4. **`x-ratelimit-remaining`/`x-ratelimit-reset` заголовки нигде не читаются.**
   Докстринг `rate_limiter.py:10` утверждает: *"API отслеживает через headers:
   x-ratelimit-limit, x-ratelimit-remaining, x-ratelimit-reset"* — но это
   описание поведения самого Stalcraft API, не код. Grep по
   `response.headers` в `backend/app/services/collector/client.py` ничего не
   находит — `_request()` не читает эти заголовки ни при обычных ответах, ни
   при 429. Узнаём о приближении к лимиту только постфактум, по факту самого
   429, а не проактивно.

5. **Совпадение по времени с коммитом `ccefafe`** (параллелизация
   `collect_all_history` на `HISTORY_CONCURRENCY=6`, `backend/app/tasks/collectors.py:20,145-167`),
   задеплоенным непосредственно перед инцидентом. Проверено: вклад этого
   изменения в сам бёрст ограничен — `_collect_history_for_item` делает только
   **один** запрос `/history` на пару (без пагинации), так что 6 параллельных
   чанков увеличивают одновременность, но не превращают обычный сбор в
   постраничный шторм запросов. Объём токенов, потребляемых обычным сбором за
   минуту, не изменился (комментарий в самом коммите: "Количество запросов к
   Stalcraft API не меняется — rate limiter централизован"). Тем не менее, если
   почасовой `collect_all_history` стартовал в то же окно времени, что и ручной
   запуск backfill, оба процесса синхронно тянут токены из ОДНОГО общего
   бакета — это могло обнулить резерв чуть раньше, чем при изолированном
   запуске backfill, но не является первопричиной: даже без пересечения с
   `collect_all_history` backfill сам по себе (пункты 1-3) гарантированно
   исчерпывает бакет на первой "толстой" паре и затем не восстанавливается
   из-за отсутствия retry текущей страницы.

**Вывод:** первопричина — отсутствие троттлинга и retry-with-same-offset внутри
самого backfill-скрипта (пункты 1-3), а не баг в общем rate limiter'е или в
недавней параллелизации сбора. Минимальный фикс ограничен ИСКЛЮЧИТЕЛЬНО файлом
`backend/app/scripts/backfill_sales_qlt.py` — он одноразовый, не часть
регулярного Celery-сбора, поэтому правки в нём не подпадают под правило
CLAUDE.md (Блок 4) о подтверждении изменений частоты опроса в продукционном
сборе.

## Изменения по слоям

### Backend (обязательная часть — только `backfill_sales_qlt.py`)

**1. Пауза между постраничными запросами внутри `_backfill_pair()`.**

В цикле `while True` (строки 142-196), после успешного запроса и до следующей
итерации (после `offset += HISTORY_PAGE_LIMIT`, либо в начале итерации перед
запросом — на выбор backend-dev, важно — после первого запроса, не до него),
добавить:

```python
await asyncio.sleep(BACKFILL_PAGE_DELAY)
```

Добавить константу рядом с существующими `HISTORY_PAGE_LIMIT`/`HISTORY_REQUEST_COST`
(строки 35-36):

```python
BACKFILL_PAGE_DELAY = 1.0  # секунд между страницами одной пары — не бёрстить
                             # резерв token bucket на полной сетевой скорости
```

Обоснование величины: `HISTORY` стоит 2 токена за запрос, устойчивый темп
бакета — 400/60 ≈ 6.67 токена/сек ⇒ один backfill-поток с паузой 1с между
запросами потребляет 2 токена/сек — это даёт большой запас на параллельную
работу обычного Celery-сбора в том же бакете (который в среднем потребляет
~54.5 запросов/мин по документации CLAUDE.md, Блок 4) и исключает локальный
бёрст на полной сетевой скорости (~5 запросов/сек, как было). Это не требует
подтверждения пользователя, т.к. НЕ меняет частоту опроса регулярного сбора —
троттлинг затрагивает только разовый скрипт.

**2. Retry той же страницы (того же `offset`) при 429, вместо проброса
исключения, которое роняет всю пару.**

Сейчас исключение `RuntimeError("Rate limit exceeded (429)")` из
`client.py::_request()` (строка 48) пробрасывается из
`stalcraft_client.get_auction_history(...)` (вызов на строке 143-145) прямо в
`run()`, где ловится `except Exception` на уровне всей пары. Нужно поймать его
**внутри `_backfill_pair()`**, на уровне одной страницы, и повторить запрос с
тем же `offset` после паузы — не идти на следующую страницу и не ронять пару.

Изменить цикл (строки 142-150) так, чтобы запрос к API был обёрнут в
retry-обёртку с ограниченным числом попыток на одну страницу:

```python
BACKFILL_MAX_PAGE_RETRIES = 5  # повторов одной страницы при 429, прежде чем сдаться

...

while True:
    for attempt in range(BACKFILL_MAX_PAGE_RETRIES):
        try:
            data = await stalcraft_client.get_auction_history(
                item_id, region=region, offset=offset, limit=HISTORY_PAGE_LIMIT,
            )
            break
        except RuntimeError as e:
            if "429" not in str(e) or attempt == BACKFILL_MAX_PAGE_RETRIES - 1:
                raise
            logger.warning(
                f"{item_id}/{region}: 429 на странице offset={offset}, "
                f"повтор {attempt + 1}/{BACKFILL_MAX_PAGE_RETRIES} после паузы"
            )
            # client.py уже спит 60с внутри _request() перед рейзом — здесь
            # дополнительной паузы не нужно, повторяем сразу после возврата.
    requests_spent += 1
    ...
```

Важно: `client.py::_request()` уже делает `await asyncio.sleep(60)` ПЕРЕД
рейзом `RuntimeError` (строка 47) — то есть пауза перед повтором уже встроена
в существующий (нетронутый) код клиента. Backend-dev должен явно это учесть и
не добавлять вторую избыточную паузу здесь — иначе повтор одной страницы при
устойчивом 429 займёт неоправданно долго (5 попыток × 60с = 5 минут на одну
страницу в худшем случае, что приемлемо для разового ночного скрипта, но
дублировать паузу не нужно).

Если после `BACKFILL_MAX_PAGE_RETRIES` попыток страница так и не прошла —
исключение пробрасывается дальше как раньше (пара помечается `failed`, переход
к следующей), но теперь это происходит только в действительно патовой
ситуации, не при первом же 429.

**3. (Опционально, не блокирует основной фикс) Проактивное чтение
`x-ratelimit-remaining` из ответа.**

Stalcraft API, по докстрингу `rate_limiter.py:10`, должен отдавать заголовки
`x-ratelimit-limit`, `x-ratelimit-remaining`, `x-ratelimit-reset` в ответах. Ни
один существующий лог/докстринг в коде не подтверждает РЕАЛЬНОЕ присутствие
этих заголовков в живых ответах (комментарий описывает ожидаемое поведение API
по документации Stalcraft, не результат проверки `response.headers` в проде) —
это нужно подтвердить эмпирически перед тем как полагаться на них.

Если backend-dev добавляет диагностику — НЕ менять `client.py` (общий для
продукционного сбора). Вместо этого: `_backfill_pair()` уже получает `dict`
от `stalcraft_client.get_auction_history(...)`, а не raw `httpx.Response` —
заголовки текущим контрактом клиента не передаются вызывающему коду. Доступа к
заголовкам без изменения сигнатуры `client.py::_request()` НЕТ. Поэтому этот
пункт реализуем ТОЛЬКО как отдельное диагностическое расширение, явно
вынесенное в раздел ниже — не часть минимального фикса.

Не затрагиваются — чисто backend-скрипт без UI.

## Требует отдельного подтверждения пользователя (НЕ включать в реализацию без явного OK)

Эти изменения затрагивают `client.py`/`rate_limiter.py`, общие для ВСЕГО
продукционного сбора (`collect_all_active_lots`, `collect_all_history` и
другие периодические Celery-таски) — по правилу CLAUDE.md (Блок 4) любые
изменения, влияющие на частоту/паттерн опроса Stalcraft API, требуют
отдельного разрешения пользователя ДО реализации:

- **Прокинуть `response.headers` наружу из `client.py::_request()`** (например,
  вернуть `tuple[dict, httpx.Headers]` вместо просто `dict`, либо завести
  `self._last_response_headers` как побочный атрибут) — чтобы вызывающий код
  (включая, но не только, backfill-скрипт) мог проактивно реагировать на
  `x-ratelimit-remaining`. Любое изменение сигнатуры `_request()` затрагивает
  ВСЕ вызовы `get_auction_lots`/`get_auction_history`/`get_emission` во всех
  Celery-таски — требует регрессионной проверки, не точечная правка.
- **Добавить retry той же страницы при 429 в сам `client.py::_request()`**
  (а не только в backfill-скрипте) — изменило бы поведение обычного сбора при
  429 (сейчас просто рейзит после 60с сна) для ВСЕХ вызывающих, не только
  backfill. Потенциально полезно (тот же тип бага мог бы проявиться и в
  обычном сборе при будущем росте watchlist), но это отдельная задача с
  отдельным анализом влияния на `collect_all_active_lots`/`collect_all_history`,
  их `max_retries`/`countdown` в Celery (`bind=True, max_retries=3`,
  `self.retry(exc=exc, countdown=60/120)`), и текущий fire-and-forget паттерн
  ошибок (`logger.error` + переход к следующему `entry`, без проброса наверх) —
  смешивать с минимальным фиксом backfill-скрипта не нужно.
- **Минимальный интервал между acquire() в `TokenBucketRateLimiter`** (сглаживание
  всплесков, не только среднего расхода) — например, не отдавать токены чаще
  раза в N миллисекунд независимо от резерва. Меняет поведение лимитера для
  ВСЕХ клиентов сразу (обычный сбор + любые будущие скрипты) — отдельная
  архитектурная задача, не нужна для решения текущего инцидента (фикс на
  уровне backfill-скрипта полностью устраняет наблюдаемый сценарий отказа без
  изменения общего лимитера).

Рекомендация: реализовать сначала только обязательную часть (пункты 1-2 в
`backfill_sales_qlt.py`), убедиться что повторный прогон скрипта на проде
проходит без застревания, и только затем — по отдельному запросу пользователя —
рассматривать пункты выше как самостоятельные задачи (если повторится похожий
инцидент в обычном сборе при будущем росте нагрузки).

## Открытые вопросы / требует подтверждения

- Точные значения `BACKFILL_PAGE_DELAY=1.0` и `BACKFILL_MAX_PAGE_RETRIES=5` —
  предложены исходя из устойчивого темпа бакета (2 токена/сек на HISTORY
  запрос при паузе 1с), но не протестированы на реальном объёме данных прода.
  Backend-dev может скорректировать после оценки `_print_cost_estimate()` для
  фактического набора пар без qlt на проде (если пар много и/или общий объём
  истории большой — возможно, стоит увеличить паузу, чтобы оставить больший
  запас токенов для параллельного обычного сбора, который продолжает работать
  во время backfill).
- Нужно ли перед повторным запуском backfill на проде делать это в окне, когда
  `collect_all_history` (почасовая) заведомо не выполняется одновременно —
  чисто организационная мера, не код. Можно упомянуть пользователю при передаче
  на тестирование/деплой, но не часть кода скрипта.
- Пункт 3 (чтение `x-ratelimit-*`) оставлен опциональным и не входит в
  обязательную реализацию — если пользователь хочет проактивный троттлинг по
  заголовкам, это отдельный заход с изменением контракта `client.py`
  (см. раздел «Требует отдельного подтверждения»).
