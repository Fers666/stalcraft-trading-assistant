# ТЗ: Точное определение статуса «продаётся ли предмет на аукционе» по всему каталогу master_items

## Контекст

Сейчас единственный источник статуса торгуемости предмета — поле `master_items.bind_state`
(`status.state` из EXBO GitHub-каталога). Фильтр непродаваемых в
`backend/app/api/v1/endpoints/items.py` считает «не появляется на аукционе» только предметы с
`bind_state IN {PERSONAL_ON_GET, PERSONAL_DROP_ON_GET}` (77 шт. из 2303).

Проблема (подтверждена пользователем): `bind_state` описывает **привязку**, а не **торгуемость**.
Множество предметов, помеченных как продаваемые (квестовые, валюта, чертежи, сюжетные, вендорные —
категории other/misc/supply/device), физически на аукционе не появляются, но проходят текущий
фильтр. Классификация по `bind_state` неверна по существу.

Решение (выбрано пользователем): единоразово пробить каждый `item_id` через Stalcraft API и
определить **реальное** наличие на аукционе через `total` в ответах `/history` и `/lots`, записать
точный статус в БД. Далее использовать этот статус как источник истины в каталоге вместо
эвристики по `bind_state`.

Распределение `bind_state` (для оценки объёма): PERSONAL_ON_USE 1387, NON_DROP 557, NONE 274,
PERSONAL_DROP_ON_GET 51, PERSONAL_ON_GET 26, NULL 8. Всего 2303.

Это **разовый бэкфилл** 2303 предметов, а не непрерывный опрос. Параметры регулярного сбора
(`LOTS_PER_RUN`/`BATCH_SIZE`/`REFRESH_INTERVAL`) не меняются.

## Исследование (выполнено)

### API-клиент
`backend/app/services/collector/client.py` → `StalcraftClient`:
- `get_auction_history(item_id, region, offset=0, limit=200)` → `GET /{region}/auction/{item_id}/history`,
  cost `TokenCost.HISTORY = 2`. Ответ: `{"total": <int>, "prices": [...]}`.
- `get_auction_lots(item_id, region, offset=0, limit=200, redis_client=None)` → `GET /{region}/auction/{item_id}/lots`,
  cost `TokenCost.LOTS = 2`. Ответ: `{"total": <int>, "lots": [...]}`.
- Авторизация — Bearer-токен через `token_manager` (OAuth, автосброс при 401). 429 → `RuntimeError`.
- Регион берётся из `settings.stalcraft_region` (сейчас `"RU"`). В коллекторах используется
  per-entry `entry.region`, но для бэкфилла каталога логично брать один регион приложения.
  **Важно:** точное имя ключа тотала (`total`) подтвердить по живому ответу при реализации —
  миграция 0022 прямо ссылается на «history_total=0 / lots_total=0», т.е. поле в ответе есть;
  считать `int(data.get("total", 0))`.

### Rate limiter
`backend/app/core/rate_limiter.py` → глобальный Token Bucket в Redis (Lua, атомарно), лимит
`CAPACITY = 400` запросов/мин, `REFILL_RATE = 400/60`. `acquire(cost, max_wait=60, redis_client=None)`
ждёт при нехватке токенов (не дольше `max_wait`), при недоступности Redis — in-memory fallback.
Непрерывная нагрузка сейчас ~54.5 req/мин (13.6%), резерв ~345.5 req/мин (86.4%).
Клиент уже вызывает `rate_limiter.acquire(cost=...)` внутри `_request` — **бэкфилл автоматически
проходит через общий лимитер и не может пробить 400/мин**. Дополнительно нужен self-throttle
(явная задержка между предметами), чтобы бэкфилл не выедал весь резерв и не заставлял непрерывный
коллектор ждать.

### Celery
`backend/app/tasks/celery_app.py` — beat + `include=[collectors, cleanup, analyzers, tiers]`,
`worker_concurrency=2`, очередь `collector`. Паттерн async-в-Celery: `run_async(coro)` +
`get_celery_db_session()` (см. `collect_all_active_lots`).

### Каталог/фильтр
`items.py` L17, L81-82 — `_UNTRADABLE_BIND_STATES = {PERSONAL_ON_GET, PERSONAL_DROP_ON_GET}`,
фильтр `or_(bind_state IS NULL, bind_state NOT IN untradable)`.
`bind_state` заполняется из GitHub в `services/catalog/github_parser.py` (остаётся как метаданные).

## Изменения по слоям

### Backend

#### 1. Схема БД (миграция `0036_master_items_on_auction`)

Добавить в `master_items` 4 nullable-поля:

| Поле | Тип | Смысл |
|---|---|---|
| `on_auction` | `Boolean`, nullable, default NULL | NULL = ещё не проверено, TRUE = торгуется, FALSE = не появляется на аукционе |
| `auction_checked_at` | `timestamptz`, nullable | Момент последней проверки через API (для resumable + periodic re-check) |
| `history_total` | `Integer`, nullable | Последний замер `total` из `/history` (аудит/отладка) |
| `lots_total` | `Integer`, nullable | Последний замер `total` из `/lots` (аудит/отладка) |

Индекс: `CREATE INDEX ix_master_on_auction ON master_items (on_auction);` — фильтр каталога будет
идти по этому полю.

**Обоснование:**
- `on_auction` nullable-boolean, а не boolean-default-false — даёт **resumability**: задача
  обрабатывает `WHERE on_auction IS NULL`, повторный запуск не перепроверяет уже пройденные.
  Различие «не знаю» (NULL) и «точно нет» (FALSE) критично: транзиентная ошибка API не должна
  выглядеть как «не торгуется».
- `auction_checked_at` нужен для необязательного периодического ре-чека (по возрасту записи) и для
  наблюдаемости прогресса.
- `history_total`/`lots_total` — дешёвая отладочная избыточность (по 4 байта): позволяют вручную
  проверить пограничные решения и не гонять API повторно при разборе спорных предметов.
  Если пользователь сочтёт лишним — можно оставить только `on_auction` + `auction_checked_at`
  (см. открытые вопросы).

Модель `MasterItem` в `models.py` — добавить те же 4 колонки и индекс в `__table_args__`.

#### 2. Критерий классификации

Порядок проверки на один предмет (экономия запросов — `/history` первым):

```
1. history = get_auction_history(item_id, region)   # cost 2
   history_total = int(history.get("total", 0))
   если history_total > 0:
       on_auction = TRUE            # был хоть раз продан → торгуется (в т.ч. редкие/сезонные)
       lots_total = NULL (не запрашивали)
       СТОП
2. иначе (history_total == 0):
   lots = get_auction_lots(item_id, region)          # cost 2
   lots_total = int(lots.get("total", 0))
   если lots_total > 0:
       on_auction = TRUE            # новый предмет, ещё без продаж, но выставлен
   иначе:
       on_auction = FALSE           # ни истории, ни лотов → не появляется на аукционе
```

**Почему `/history` важнее и первым:** редкий/сезонный предмет может иметь 0 активных лотов
прямо сейчас, но ненулевую историю продаж → это торгуемый предмет. Текущие лоты вторичны и нужны
только чтобы поймать «новый предмет, выставлен, ещё не продан». Проверка `/history` первой экономит
запросы: у торгуемых предметов (их большинство) хватит одной пары запросов (cost 2), вторая пара
(`/lots`) тратится только на кандидатов в «не торгуется».

**Пограничные случаи:**
- **404 от API** (предмет не существует в системе аукциона / удалён из игры) → `on_auction = FALSE`,
  `history_total = lots_total = 0`, `auction_checked_at = now`. Это валидный отрицательный ответ.
- **Транзиентная ошибка** (5xx, таймаут, сетевой сбой, `TimeoutError` от лимитера, неожиданный 429)
  → **НЕ писать FALSE**. До 2 ретраев с backoff (напр. 2s, 5s). Если не удалось — оставить
  `on_auction = NULL`, залогировать `item_id` + ошибку; предмет попадёт в следующий прогон.
- **401** — обрабатывается внутри клиента (автосброс токена), наружу не всплывает.
- Идемпотентность: запись `on_auction`/`*_total`/`auction_checked_at` через UPDATE по `item_id`,
  повторный проход того же предмета даёт тот же результат.

#### 3. Механизм запуска — Celery-задача + admin-триггер (рекомендация)

Рекомендуемый подход: **Celery-задача `audit_auction_status`, запускаемая admin-эндпоинтом**.

- Задача (в `tasks/collectors.py` рядом с остальными коллекторами, либо новый `tasks/audit.py`):
  - Параметры: `force_recheck: bool = False`, `stale_days: int | None = None`,
    `limit: int | None = None` (для тестового прогона на N предметов).
  - Выборка предметов:
    - обычный прогон: `WHERE on_auction IS NULL` (resumable — только непроверенные);
    - `force_recheck=True`: все предметы;
    - `stale_days=N`: `WHERE on_auction IS NULL OR auction_checked_at < now() - N days`
      (для периодического ре-чека).
  - Обрабатывает предметы **последовательно**, коммит после **каждого** предмета (устойчивость к
    рестарту воркера — прогресс не теряется).
  - Self-throttle: `await asyncio.sleep(AUDIT_REQUEST_DELAY)` между предметами (см. бюджет ниже).
  - Один общий `redis_client` на прогон (как в `collect_all_active_lots`), передаётся в
    `get_auction_lots(..., redis_client=...)`; для `get_auction_history` клиент не
    прокидывается — приемлемо (лимитер создаст своё соединение).
  - Логирование прогресса: каждые ~50 предметов — `INFO` с `checked/total`, счётчиками
    TRUE/FALSE/errors.
  - `bind=True, max_retries=0` (ретраи — внутренние, на уровне одного предмета, чтобы падение на
    одном предмете не рестартовало всю задачу).
- Admin-эндпоинт `POST /api/v1/admin/audit-auction-status` (`get_current_admin`): ставит задачу в
  очередь (`audit_auction_status.delay(...)`), возвращает `{status: "queued", task_id}`.
  Дополнительно `GET /api/v1/admin/audit-auction-status` — прогресс из БД:
  `total`, `checked` (on_auction NOT NULL), `on_auction_true`, `on_auction_false`, `pending` (NULL),
  `last_checked_at`.

**Почему так, а не management-скрипт / синхронный эндпоинт:**
- Переиспользует существующую инфраструктуру (Celery worker, `get_celery_db_session`, `run_async`,
  общий rate limiter) — минимум нового кода.
- Долгий прогон (десятки минут, см. оценку) нельзя держать в HTTP-запросе — Celery выполняет в
  фоне, эндпоинт отвечает мгновенно.
- Resumable «из коробки» через `WHERE on_auction IS NULL` + покоммитный прогресс: рестарт воркера/
  деплой не начинает заново.

**Периодический ре-чек (опционально, НЕ обязательно):** предметы появляются/исчезают с аукциона.
Можно добавить в beat ежемесячный запуск `audit_auction_status(stale_days=30)`. По умолчанию **не
включать** — оставить ручной запуск; решение о периодике за пользователем (см. открытые вопросы).

#### 4. Бюджет API-запросов и оценка времени (ТРЕБУЕТ ПОДТВЕРЖДЕНИЯ ПОЛЬЗОВАТЕЛЯ)

Стоимость в «запросах-единицах» лимитера (не HTTP-вызовах): торгуемый предмет = 2 (только
`/history`), неторгуемый/кандидат = 4 (`/history` + `/lots`).

Границы полного прогона 2303 предметов:
- нижняя (все торгуемые, по одной паре): `2303 × 2 = 4606` ед.;
- верхняя (все требуют обе пары): `2303 × 4 = 9212` ед.;
- реалистичная (~1500 торгуемых + ~800 нет): `1500×2 + 800×4 ≈ 6200` ед.

Резерв лимитера ~345 ед/мин, но выедать его целиком нельзя (конкуренция с непрерывным сбором).
Предлагаемый **бюджет бэкфилла ≈ 100 ед/мин** (25% от 400; вместе с текущими 54.5 → ~154 ед/мин =
~38% лимита, безопасно). Реализуется через `AUDIT_REQUEST_DELAY` между предметами:
при ~2-4 ед/предмет и цели 100 ед/мин → задержка ~1.5–2.4 с/предмет.

Оценка времени полного прогона при бюджете 100 ед/мин:
- реалистично ~6200/100 ≈ **62 минуты**;
- худший случай ~9212/100 ≈ **92 минуты** (~1.5 ч).

Более консервативный бюджет 60 ед/мин → реалистично ~103 мин, худший ~154 мин (~2.5 ч).

Лимитер жёстко гарантирует, что суммарно (сбор + бэкфилл) не превысим 400/мин; self-throttle нужен
лишь чтобы бэкфилл не «съедал» резерв и не тормозил непрерывный коллектор. Значение
`AUDIT_REQUEST_DELAY` вынести константой в задаче. **Параметры непрерывного опроса
(`LOTS_REFRESH_INTERVAL`/`MAX_LOTS_PER_RUN`/`TARGET_CYCLE_SEC`) не трогаем.**

#### 5. Смена фильтра каталога (`items.py`) — двухфазный переход

Trade-off: пока бэкфилл не прошёл полностью, часть предметов имеет `on_auction IS NULL`
(не проверены). Резко переключаться на `WHERE on_auction IS TRUE` нельзя — до завершения бэкфилла
каталог «опустеет».

Рекомендация — **двухфазно**:
- **Фаза A (сразу после миграции, во время/после бэкфилла):** фильтр показывает всё, кроме
  подтверждённо неторгуемого:
  ```
  WHERE on_auction IS NOT FALSE
    AND (on_auction IS TRUE
         OR bind_state IS NULL
         OR bind_state NOT IN {PERSONAL_ON_GET, PERSONAL_DROP_ON_GET})
  ```
  То есть `on_auction = FALSE` скрывает, `TRUE` показывает, `NULL` (ещё не проверен) — падает на
  старую эвристику `bind_state`. Каталог не ломается на переходном периоде.
- **Фаза B (после полного бэкфилла и ручной верификации, когда `pending = 0`):** чистый фильтр
  ```
  WHERE on_auction IS TRUE
  ```
  и удаление `_UNTRADABLE_BIND_STATES`-эвристики из запроса.

`bind_state` как колонка **остаётся** (метаданные из GitHub, синхронизируется каталогом), но
перестаёт быть источником статуса торгуемости. Переключение Фаза A → Фаза B — отдельный маленький
коммит после подтверждения, что бэкфилл завершён (`GET /admin/audit-auction-status` → `pending = 0`).

### Frontend

По умолчанию правок не требуется: `/items` вернёт меньше предметов (уйдёт мусор), контракт
`ItemResponse` не меняется. **Опционально** (по желанию пользователя, отдельной задачей): добавить
`on_auction` в `ItemResponse` и показывать бейдж/фильтр «только торгуемые» в каталоге. В базовый
объём не входит.

## Критерии успеха (верифицируемые)

- [ ] Миграция 0036 применяется и откатывается; в `master_items` есть 4 новых поля + индекс
      `ix_master_on_auction`; все существующие строки имеют `on_auction IS NULL`.
- [ ] Модель `MasterItem` синхронна со схемой (4 поля).
- [ ] Задача `audit_auction_status` зарегистрирована в Celery (`include`), запускается через
      `POST /api/v1/admin/audit-auction-status` (только админ), возвращает `task_id`.
- [ ] Логика классификации соответствует таблице: history>0 → TRUE (без вызова /lots);
      history=0 & lots>0 → TRUE; оба 0 → FALSE; 404 → FALSE; транзиентная ошибка → остаётся NULL.
- [ ] Resumable: повторный запуск обрабатывает только `on_auction IS NULL`; прерывание/рестарт
      воркера не теряет прогресс (покоммитно) и не начинает заново.
- [ ] Идемпотентность: два прогона подряд на одном предмете дают одинаковый результат.
- [ ] Self-throttle держит бэкфилл в пределах согласованного бюджета (≈100 ед/мин); при работающем
      бэкфилле непрерывный сбор лотов не деградирует (нет всплеска 429, `get_consumption_stats`
      суммарно < 400/мин).
- [ ] `GET /api/v1/admin/audit-auction-status` отдаёт корректные `total/checked/true/false/pending`.
- [ ] Полный прогон завершается: `pending = 0` (кроме предметов с устойчивой ошибкой API —
      залогированы поimённо).
- [ ] После полного прогона `/items` (Фаза A) не показывает предметы с `on_auction = FALSE` и не
      теряет предметы с `on_auction = TRUE`; выборочная ручная проверка 5–10 квестовых/валютных
      предметов → они исчезли из каталога, торгуемые остались.
- [ ] (Фаза B, после верификации) фильтр `on_auction IS TRUE`, `bind_state`-эвристика удалена из
      запроса `list_items`.
