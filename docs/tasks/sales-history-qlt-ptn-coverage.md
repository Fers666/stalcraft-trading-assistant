# ТЗ: Повысить покрытие qlt/ptn в sales_history

## Контекст

Задача из backlog (`docs/NOTES.md`): текущее покрытие `qlt`/`ptn` (качество и
заточка артефакта) в `sales_history.additional_info` критически низкое —
изначально оценено в 2-6%, в локальной БД на момент исследования — 12%
(8 636/71 811).

### Первая версия этого ТЗ содержала неверный диагноз — зафиксировано для прозрачности

Первая версия документа исходила из предположения (взятого из комментария в
коде `collectors.py:615`, дословно: *«Stalcraft API /history не возвращает
additional с qlt/ptn»*) и предлагала решать проблему через снэпшот-матчинг
(новую таблицу `lot_identity` или расширение окна снэпшотов). **Это
предположение проверено и опровергнуто прямым запросом к Stalcraft API через
рабочий backend-контейнер.**

Живая проверка (`stalcraft_client.get_auction_history(...)`, read-only):

```
data = await stalcraft_client.get_auction_history('qoq6', region='RU', offset=0, limit=10)
```
→ каждая запись уже содержит `additional.qlt` (и часто `ptn`) напрямую:
```python
{'amount': 1, 'price': 19000000, 'time': '2026-06-28T21:40:41Z',
 'additional': {'it_transf_count': 1, 'qlt': 4, 'upgrade_bonus': 0.0, 'spawn_time': 1782673206321}}
{'amount': 1, 'price': 1770000, 'time': '2026-06-28T21:40:03Z',
 'additional': {'bonus_properties': [...], 'qlt': 2, 'ptn': 15, ...}}
```
Проверено также на глубокой странице (`offset=1990`, ~3 дня назад) — `qlt`
присутствует одинаково надёжно, включая явный `qlt: 0` (не просто отсутствие
поля). Для неартефактных предметов (оружие, например `m02wr`) `additional` не
содержит `qlt`/`ptn` — это ожидаемо (у оружия нет качества/заточки в этом
смысле), не баг и не повод для fallback-логики.

**Корневая причина проблемы — не отсутствие данных в API, а то, что код их
игнорирует.** В `_collect_history_for_item()` (`collectors.py`, цикл `for
record in prices:`, строки ~664-708) из каждой записи `/history` читаются
только `record.get("time")`, `record.get("price")`, `record.get("amount")`.
`record.get("additional")` **нигде не читается** — подтверждено отсутствием
любых упоминаний в файле. При этом запрос к API уже идёт с параметром
`additional: "true"` (`client.py::get_auction_history`, строка 64) — то есть
поле запрашивается, приходит в ответе, но молча отбрасывается на этапе
парсинга.

Вместо этого код пытается **реконструировать** qlt/ptn косвенно — через
сопоставление проданного лота с тем же лотом, который ранее был виден в
снэпшоте `/lots` (см. `find_lot_info()`, окно `SNAPSHOT_MATCH_WINDOW_HOURS=2`,
≈1.7ч по 200 последним снэпшотам). Эта реконструкция работает лишь для узкого
окна и объясняет наблюдаемое низкое покрытие — но она оказалась полностью
ненужной: то же самое поле уже лежит прямо в ответе `/history`, без всякого
матчинга.

### Что нужно вместо этого

Минимальное точечное изменение: в цикле обработки `record` читать
`record.get("additional")` и использовать его **как первичный, authoritative
источник** qlt/ptn (пришло прямо с фактом продажи от API — точнее и надёжнее
любой реконструкции). Существующий снэпшот-матчинг (`find_lot_info`)
сохраняется как **вторичный fallback** — но его практическая польза теперь
видится исключительно для `lot_start` (время выставления лота, нужно для
`time_on_market = sale_time - lot_start`), не для qlt/ptn. Если `additional`
от API пуст (оружие/броня без качества) — это корректный случай, не
обрабатывается как ошибка.

`lot_identity` (отдельная таблица для предварительного накопления identity
лотов из снэпшотов) и расширение `SNAPSHOT_MATCH_WINDOW_HOURS` — **больше не
нужны для решения проблемы покрытия qlt/ptn** и не включаются в эту итерацию.
Они МОГЛИ БЫ иметь смысл для отдельной, менее приоритетной задачи —
повышения точности/покрытия `lot_start` (time-on-market), но это не предмет
текущего ТЗ и не запрашивалось.

## Проверено в БД (read-only, dev-окружение)

- `sales_history`: всего 71 811 строк, без `qlt` — **63 175** (88%), затронуто
  **31** уникальная пара `(item_id, region)`.
- Глубина истории API существенно больше, чем что-либо нужное для текущего
  TTL: для предмета `qoq6` (`RU`) `total: 652 280` записей в `/history`
  (поле `total` в ответе). Offset→время по тому же предмету:
  - `offset=0` → 2026-06-28 (сейчас)
  - `offset=5 000` → 2026-06-22 (6 дней)
  - `offset=20 000` → 2026-06-05 (23 дня)
  - `offset=50 000` → 2026-04-30 (60 дней)
  - Линейно экстраполируя — нужный `offset` для покрытия текущего TTL
    `sales_history` (120 дней) для этого предмета составляет
    **≈100 000+** записей, то есть **≈500-700 запросов** по `limit=200` за
    раз (с `cost=2` за запрос на `/history` → **≈1000-1400 токенов rate
    limit на один предмет** для полного backfill глубиной 120 дней). Это
    варьируется сильно по объёму продаж конкретного предмета — для менее
    ликвидных позиций глубина в записях на тот же временной диапазон будет
    меньше.

Это ключевая цифра для оценки стоимости backfill — см. открытые вопросы.

## Изменения по слоям

### Backend

**1. `_collect_history_for_item()` — основной фикс**

В цикле `for record in prices:` (после получения `total_price`, `amount`,
`sold_at`):

```python
api_additional = record.get("additional") or {}
```

Использовать `api_additional` как основу для `lot_info`, дополняя (не
заменяя) результатом `find_lot_info()` только то, чего не хватает (в первую
очередь `lot_start`, которого в `/history` нет):

- Для **новой записи**: `lot_info = dict(api_additional)`; если в
  `api_additional` нет `qlt`/`ptn` (предмет без качества) — оставить как
  есть, не запускать снэпшот-матчинг ради qlt/ptn специально, но **продолжать
  вызывать** `find_lot_info()` для получения `lot_start` (если
  `sold_at >= snapshot_cutoff`, как и сейчас) и смержить результат: `lot_info
  = {**snapshot_lot_info, **api_additional}` (API-данные приоритетнее, если
  пересекаются по `qlt`/`ptn`).
- Для **существующей записи без qlt** (ретроактивное докручивание, текущий
  блок `needs_qlt`): заменить логику — раньше она проверяла `sold_at >=
  snapshot_cutoff` (то есть докручивала только недавние продажи, т.к.
  зависела от окна снэпшотов). Теперь, поскольку источник — сам ответ
  `/history` (не зависит от снэпшота), это ограничение по времени **больше не
  нужно для qlt/ptn**: если `record.get("additional")` содержит `qlt` —
  обновить `additional_info` независимо от возраста продажи. Это автоматически
  частично выполняет работу "backfill" в рамках уже существующего часового
  прохода `collect_all_history` — но **только для продаж, которые ещё
  попадают в окно `/history`, которое реально опрашивается** (нужно сверить
  текущий диапазон `offset`/`limit`, который запрашивает текущий код — если
  опрашивается только `offset=0, limit=200`, это покроет лишь последние ~200
  продаж за час, не весь пробел).

**2. Исправить комментарий** на строке ~615 (`find_lot_info`) — убрать
утверждение «Stalcraft API /history не возвращает additional с qlt/ptn»,
заменить на корректное описание (API возвращает, но матчинг по снэпшотам
нужен только для `lot_start`).

**3. Backfill-скрипт** (разовый, не периодическая задача) — **решение
пользователя зафиксировано**: CLI-скрипт `backend/app/scripts/backfill_sales_qlt.py`,
запуск вручную через `docker compose exec backend python -m
app.scripts.backfill_sales_qlt --days 30` (без admin-эндпоинта/кнопки в UI).
`--days` — флаг с default=30 (можно перезапустить позже с большей глубиной
по отдельному запросу, см. п.2 открытых вопросов).

- Выбрать `sales_history` записи без `qlt` (`additional_info` не содержит
  ключ `qlt`, или `additional_info IS NULL`) **и `sale_time >= now() -
  interval '{days} days'`**, сгруппировать по `(item_id, region)`.
- Для каждой пары — постранично запрашивать `/history` с нарастающим `offset`
  (`limit=200`), пока не покрыт диапазон `--days` или ответ не опустеет.
- Сматчить полученные записи API с существующими строками `sales_history` по
  `(sale_time, total_price, amount)` (тот же ключ, который уже используется
  в `existing_index` текущего кода) — найдя совпадение, обновить
  `additional_info` из `record.get("additional")`.
- **Важно про rate limit**: это съедает реальные токены `/history` (cost=2
  каждый). Использовать тот же `stalcraft_client`/Redis token bucket, что и
  обычный сбор — не отдельный механизм троттлинга. Скрипт должен логировать
  прогресс (сколько пар обработано, сколько токенов rate limit израсходовано)
  — выполняется вручную из терминала, разового мониторинга через логи
  достаточно, без необходимости в фоновой Celery-задаче.

1. ~~Backfill — когда и как запускать.~~ **Решено пользователем**: разовый

   API-запросов (read-only `total` по затронутым парам) в лог/консоль, чтобы

2. ~~Глубина backfill~~ **Решено пользователем**: 30 дней по умолчанию (флаг
   `--days`, можно перезапустить позже с большей глубиной отдельным запросом).
3. **Ретроактивное докручивание в рамках обычного часового
   `collect_all_history`** (см. п.1 раздела Backend) — нужно уточнить, какой
   диапазон `offset/limit` сейчас реально запрашивается у `/history` в этой
   функции (видимо `offset=0, limit=200` по умолчанию — нужно явно
   проверить в реализации перед тем, как закладывать ожидания по покрытию
