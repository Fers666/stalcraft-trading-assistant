# ТЗ: «Радар рынка» — кросс-юзерная агрегация watchlist

## Контекст

Следующий пункт роадмапа подписок (`docs/NOTES.md`, фаза «Роадмап подписок»):
«Радар рынка» — кросс-юзерная агрегация watchlist пользователей, отдельный
аддон, не привязанный к тарифам.

**Watchlist уже существует и полностью реализован** (это ключевая находка —
сложность задачи ниже, чем если бы пришлось проектировать механизм
отслеживания с нуля):

- Модель `UserWatchlist` (`backend/app/models/models.py:88`) — `user_id`,
  `item_id`, `region`, `quality_filter`, `enchant_filter`, `is_active`,
  `created_at`.
- CRUD-эндпоинты `backend/app/api/v1/endpoints/watchlist.py` (`GET/POST/PUT/
  DELETE /watchlist/`, `POST /watchlist/{id}/refresh`).
- Тарифные лимиты на размер watchlist (`backend/app/core/tiers.py`) — 6/10/20/25
  карточек по тарифу, без лимита для админов.
- Сбор данных по watchlist уже дедуплицирован across users на уровне Celery
  (`collect_all_active_lots`, см. `docs/SERVICES.md` «Логика дедупликации
  watchlist») — `unique_pairs = {(item_id, region) for e in all_active_watchlist}`.
  Это значит, что данные для агрегации физически уже собираются с частотой
  раз в ~20 сек на уникальную пару `(item_id, region)`, независимо от
  количества пользователей, отслеживающих её — **агрегация «Радара» не
  добавляет новых обращений к Stalcraft API**, она просто читает существующие
  таблицы `user_watchlist` + `collected_data` + `market_statistics`.
- **Поле-заготовка уже существует**: `User.has_market_radar_addon`
  (`backend/app/models/models.py:29`, миграция `0026_user_tiers.py`) — boolean,
  default `false`, добавлено явно «под будущую фазу, без логики в этой
  миграции» (см. docstring миграции и `docs/DATABASE.md` §`users`). Это прямое
  архитектурное указание от предыдущей итерации: аддон задуман как **отдельный
  флаг пользователя, не как часть `tier`** — подтверждает трактовку «не
  привязан к уровню тарифа» из формулировки задачи.
- Аналогичный паттерн гейтинга уже есть для аукциона: `TierLimits.auction_access`
  (тарифный, не аддон) — отличается от `has_market_radar_addon` тем, что
  аукцион гейтится по `tier`, а Радар — по отдельному boolean. Это
  единственный прецедент гейтинга в проекте, оба будут жить рядом в
  `UserResponse`/`Layout.tsx` по одной и той же визуальной схеме (`gated` +
  `lockSvg` + tooltip, см. `frontend/src/components/Layout.tsx:94-118`).

## Архитектурное решение

### 1. Гейтинг доступа — отдельный аддон-флаг (не часть тарифной матрицы)

**Выбор:** `User.has_market_radar_addon` (уже существует) — независимый
boolean, проверяется отдельным dependency `get_market_radar_access`, по
аналогии с `get_current_admin` (`backend/app/core/dependencies.py:55-58`).
Админы (`is_admin=True`) обходят флаг — единый паттерн с `get_tier_limits`.

**Почему не часть `TierLimits`:** `TIERS` в `tiers.py` — дискретная лестница
4 уровней с понятной прогрессией лимитов одного типа (карточки/окна/доступ).
Радар рынка концептуально другое измерение — не «сколько ты можешь
отслеживать», а «видишь ли агрегат по всем пользователям». Привязка к tier
заставила бы либо включить его во все тарифы сразу (бессмысленно для `base`,
если задумана монетизация), либо создать 5-й уровень-фикцию. Отдельный флаг
даёт точечную монетизацию (продать аддон пользователю `base`, не повышая
основной тариф) — что и зафиксировано в миграции 0026 заранее.

**Trade-off, который принимаем:** два независимых места выдачи доступа
(`tier` и `has_market_radar_addon`) в админке вместо одной кнопки — чуть
больше UI-поверхности, но соответствует природе аддона как доп. покупки.

**Что не делаем:** не вводим биллинг/оплату аддона в этой задаче — только
ручную выдачу флага админом (как сейчас `tier` выдаётся вручную через
`POST /admin/users/{id}/tier`). Платежи — отдельная нереализованная пока тема
во всём проекте (тарифы тоже выдаются вручную, не через эквайринг).

### 2. Где считается агрегация — on-the-fly SQL-запрос, без новой Celery-задачи

**Вариант A (выбран): on-the-fly агрегирующий SQL-запрос по `user_watchlist`**
при каждом GET-запросе страницы, с коротким Redis-кэшем (TTL 60 сек).

**Вариант B (отклонён): отдельная Celery-задача + материализованная таблица**
`market_radar_snapshot`, пересчитываемая раз в N минут — по аналогии с
`market_statistics`.

**Почему A, а не B:**
- Источник данных — `user_watchlist` (счётчики GROUP BY) — это лёгкий запрос
  (один индекс `ix_watchlist_active(user_id, is_active, region)`, таблица
  держит десятки-сотни строк на реалистичном масштабе проекта, не миллионы).
  GROUP BY по `item_id, region` с `COUNT(DISTINCT user_id)` не требует
  предрасчёта — это не то же самое по стоимости, что `market_statistics`
  (которая агрегирует тысячи строк `sales_history`/`collected_data` на
  предмет, поэтому там оправдан часовой пересчёт).
- Новая Celery-задача — это новая периодическая нагрузка на воркер и ещё одна
  точка отказа/синхронизации, ради запроса, который не бьёт по Stalcraft API
  (агрегация только над собственной БД, не вызывает `/lots` или `/history`).
  **Не затрагивает rate limit Stalcraft вообще** — не требует подтверждения
  пользователя по этому пункту (не трогаем `LOTS_PER_RUN`/`BATCH_SIZE`/
  `REFRESH_INTERVAL`, не добавляем новых обращений к внешнему API).
- Короткий Redis TTL (60 сек) защищает от повторного пересчёта при частых
  заходах на страницу несколькими пользователями одновременно — простой кэш
  без отдельной таблицы и без миграции схемы.

**Что не делаем:** не создаём новую таблицу/модель. Используем только
существующие `user_watchlist`, `master_items`, `market_statistics`
(последняя — чтобы подмешать текущую цену/объём к топу предметов, не считая
её повторно).

**Если нагрузка вырастет на порядки** (тысячи активных пользователей) —
вариант B остаётся резервным планом, отмечается в открытых вопросах ниже как
то, что стоит пересмотреть при росте, но не нужно проектировать заранее
(YAGNI).

### 3. Набор метрик «Радара»

Из того, что физически есть в БД без новых полей:

1. **Топ отслеживаемых предметов** — `COUNT(DISTINCT user_id)` по
   `user_watchlist` где `is_active=true`, GROUP BY `item_id` (без разбивки по
   region/quality/enchant — это фильтры конкретного пользователя, для
   community-агрегации избыточны), JOIN `master_items` для имени/иконки,
   ORDER BY count DESC LIMIT 20.
2. **Контекст по каждому предмету из топа** — подмешать последний снэпшот из
   `market_statistics` (global, `user_id IS NULL`): `avg_price_24h`,
   `sales_volume_24h`, при наличии — `demand_signals.bulk_spike`. Если записи
   нет (предмет ни у кого не считан) — поля `null`, фронт показывает «нет
   данных».
3. **Тренд за окно** (новое наблюдение в духе «на что сейчас смотрит рынок») —
   сравнение `COUNT` отслеживающих сейчас vs `COUNT` на момент,
   соответствующий `created_at <= now() - 24h` (т.е. сколько из текущих
   watcher'ов добавили предмет за последние 24ч). Считается тем же запросом
   через `COUNT(...) FILTER (WHERE created_at >= now() - interval '24 hours')`.
   Это даёт «прирост интереса за 24ч» без отдельной истории снэпшотов.
4. **Сводная метрика** (опционально, простая): общее число активных watcher
   across всех пользователей и число уникальных отслеживаемых предметов —
   контекст в духе «всего N карточек отслеживается M пользователями».

**Чего не делаем в Phase 1:** не показываем сами цены/лоты конкретных
пользователей (это персональные фильтры `quality_filter`/`enchant_filter` —
раскрывать их в community-агрегации избыточно и потенциально деанонимизирует
поведение пользователя при малой выборке). Агрегация исключительно на уровне
`item_id` — без user-level детализации.

## Открытые вопросы / требует подтверждения

1. **Гейтинг: «отдельный аддон» (выбрано) vs «бесплатно для всех тарифов» vs
   «часть тарифа advanced_plus+».** Формулировка задачи допускает обе
   трактовки («не привязан к тарифам» можно прочитать и как «бесплатно для
   всех», и как «отдельная платная сущность вне лестницы тарифов»). Выбрана
   вторая трактовка — она соответствует уже существующему полю
   `has_market_radar_addon` и docstring миграции 0026, которые явно говорят
   об «аддоне». **Если пользователь имел в виду «доступно всем
   пользователям независимо от тарифа бесплатно»** — реализация существенно
   проще: дополнительный дependency не нужен, достаточно `get_current_user`,
   а `has_market_radar_addon` остаётся неиспользуемым полем. Нужно подтвердить
   перед началом `backend-dev`.
2. **Способ выдачи аддона админом.** Предложен ручной toggle в админке
   (аналог тарифа) — нет биллинга/оплаты в этой задаче. Подтвердить, что это
   ожидаемый объём работы (не нужна интеграция с платежами на этом этапе).
3. **Состав метрик.** Предложен топ-20 предметов по числу watcher'ов + прирост
   за 24ч + контекст цены/объёма из `market_statistics`. Если нужны другие
   срезы (например, разбивка по регионам, фильтр по категории предмета,
   динамика за 7д вместо 24ч) — уточнить до реализации, так как это влияет на
   сложность SQL-запроса и кэш-стратегию.
4. **Видимость при пустых данных.** Если у предмета в топе нет записи в
   `market_statistics` (никто из админов/системы её не считал) — предложено
   показывать `null`/«нет данных» в UI, а не скрывать предмет из топа.
   Подтвердить, что это ожидаемое поведение.
5. **Минимальный порог анонимности.** Не вводим в Phase 1 порог «показывать
   предмет в топе только если ≥N watcher'ов» — на малом масштабе проекта
   (десятки пользователей) топ-20 может включать предметы с 1 watcher'ом,
   что фактически раскрывает «кто-то один следит за этим». Если это
   нежелательно — нужно явное решение о минимальном пороге (например, ≥3
   watcher'а для попадания в топ) до реализации.

## Ревизия: группировка по качеству/заточке (2026-06-28)

**Триггер:** ревью пользователя после реализации backend+frontend (не закоммичено).
Текущая агрегация в `market_radar.py` группирует топ только по `item_id`
(`GROUP BY UserWatchlist.item_id`, строка 84) — это теряет различие между
watcher'ами, отслеживающими один и тот же предмет с разными
`quality_filter`/`enchant_filter`. Правка: **уникальная строка топа = `(item_id,
quality_filter, enchant_filter)`**, а не только `item_id`.

### 1. Источник цены/объёма после правки

**Решение:** ветвление по наличию фильтра, без новой абстракции:

- **Бакет `quality_filter IS NULL AND enchant_filter IS NULL`** («любое
  качество/любая заточка») — источник остаётся `market_statistics` (global,
  `user_id IS NULL`), как сейчас. Это самый населённый и самый «дешёвый»
  случай — никаких изменений в эту ветку не вносим.
- **Любой бакет, где задан хотя бы один фильтр** — источник: прямой SQL по
  `SalesHistory`, переиспользуя **`_build_sales_filter(quality_filter,
  enchant_filter)`** из `backend/app/api/v1/endpoints/monitoring.py:57-78`
  (импортировать функцию, не дублировать условия). Берём `price_per_unit` за
  окно 7 дней (`sale_time >= now() - 7d`), считаем `statistics.median()` →
  цену-ориентир и `len(prices)` → объём.

  **Почему `_build_sales_filter`, а не `pricing.compute_reference`:**
  `compute_reference` — чистая функция комбинирования уже посчитанных чисел
  (`median_hist`, `median_now`, `current_min` → `{ref, trend}`), она не делает
  SQL и не знает про `quality_filter`/`enchant_filter` напрямую. Реальный SQL
  с фильтрацией по qlt/ptn находится в `monitoring.py` рядом с
  `_build_sales_filter` (строки 243-260: `select(SalesHistory.price_per_unit)
  .where(item_id, region, sale_time >= cutoff, *extra_conds)`). Это и есть
  готовый паттерн, который нужно переиспользовать «как есть» — `compute_reference`
  в данном случае не подходящий уровень абстракции (он про fast/normal/premium
  цены для уведомлений, market radar не нужен этот слой).

  **Ожидаемое и зафиксированное поведение:** при покрытии `SalesHistory.additional_info`
  qlt/ptn в 2-6% (см. `docs/NOTES.md`, известное ограничение — окно матчинга
  лота со снэпшотом ~1.7ч) для большинства бакетов с заданным фильтром
  `prices` будет пустым списком → `avg_price_24h=None`, `sales_volume_24h=None`/`0`.
  **Это не баг** — фронт показывает «нет данных», как и для случая отсутствия
  записи в `market_statistics` (п.4 исходного ТЗ). Не нужно скрывать такие
  строки из топа — `watchers_count` валиден независимо от наличия цены.

### 2. Регион

**Решение по умолчанию: агрегировать по всем регионам вместе** (не фильтровать
по `region` при запросе `SalesHistory`), сохраняя нестрогость, которая уже
есть в текущей реализации для `market_statistics`.

**Обоснование:** пользователь просил группировку только по `item_id +
quality_filter + enchant_filter` — про регион явно ничего не сказано. Текущий
код уже не разбивает топ по региону (берёт произвольную запись
`market_statistics` без фильтра по `region`, см. `_calculate_market_radar_aggregate`
строки 101-108) — это существующее поведение, ревизия его не трогает и не
делает строже задним числом. Добавление `region` в `GROUP BY` увеличило бы
число строк топа ещё сильнее (item × quality × enchant × region) без явного
запроса на это — нарушило бы Simplicity First. Если в будущем понадобится
разбивка по региону — отдельная задача с явным запросом пользователя.

**Trade-off:** для предметов, где EU/RU/SEA имеют разную ликвидность, медиана
за 7д будет «смешанной» по регионам — приемлемо для community-обзора («на что
смотрит рынок»), неприемлемо было бы для точного прогноза цены конкретному
пользователю (но это и не цель Радара — он не заменяет `/monitoring/item`).

### 3. NULL как отдельный бакет — подтверждено

`(item_id, quality_filter=NULL, enchant_filter=NULL)` — самостоятельный,
не объединяемый бакет. SQL: `GROUP BY UserWatchlist.item_id,
UserWatchlist.quality_filter, UserWatchlist.enchant_filter` — `GROUP BY` в
PostgreSQL естественным образом разделяет `NULL` как отдельное значение
группы, никакой дополнительной логики (`COALESCE`/`IS NULL` спецкейсы) не
требуется для самой группировки (только для SQL фильтрации цены, где
`_build_sales_filter` уже обрабатывает `0`/`NULL` раздельно).

Один физический предмет может занять несколько строк топ-20 одновременно
(например: «Деталь X — любое качество/заточка», «Деталь X — качество 3/любая
заточка», «Деталь X — качество 5/+12») — это прямое следствие запроса
пользователя, не побочный эффект для исправления.

### 4. Frontend-отображение

**Решение:** переиспользовать чиповый паттерн из `MonitoringPage.tsx:305-324`
(не текстовые эмодзи ⭐/⚡ из Telegram-уведомлений — тот паттерн для plain-text
сообщений бота, в React-карточке естественнее MUI `Chip`):

  ничего не показывать, как в `MonitoringPage.tsx:305`).
- Качество (`quality_filter`): `Chip` с `QLT_NAMES[quality_filter]` (уже

### 5. Влияние на лимит топ-20

**Ожидаемый побочный эффект, не требует действий сейчас:** популярный предмет
с несколькими активными комбинациями qlt/ptn среди watcher'ов может занять до
N строк топ-20 (где N — число уникальных комбинаций), вытесняя другие
предметы. `TOP_LIMIT = 20` не меняется в рамках этой ревизии — если на
практике топ окажется заполнен вариациями 2-3 популярных предметов, это
повод для отдельного решения пользователя (например, "не более K строк на
один item_id" или увеличение LIMIT) после наблюдения на реальных данных, не
для предположения заранее (YAGNI).

## Ревизия 2: метрика выгодных предложений (2026-06-28)

**Триггер:** сверка с оригинальным запросом пользователя на «Радар рынка»
выявила пропуск: пользователь просил, помимо `watchers_count`, **отдельную
метрику** «количество выгодных предложений по лоту» с явной дедупликацией —
«1 предложение для 10 пользователей это 1 предложение в статистике». Текущая
реализация показывает только `watchers_count` (сколько пользователей следит),
без подсчёта самих выгодных лотов в снэпшоте. Обе метрики остаются в строке
топа — не заменяют друг друга (подтверждено пользователем).

### 1. Источник логики оценки выгодности — переиспользование, не дублирование

`compute_signals_for_entry` (`backend/app/services/profitable_lots.py:91-255`)
уже делает ровно то, что нужно (оценка лотов снэпшота на выгодность), но
заточена на одну `UserWatchlist`-запись конкретного пользователя — масштабировать
её «как есть» на 20 строк топа означало бы пересчитывать `ref`/`sell_options`
по персональным данным каждого из потенциально многих watcher'ов на бакет,
то есть именно ту недедуплицированную работу, которую просил исключить
пользователь. Вместо вызова `compute_signals_for_entry` на каждого watcher'а —
переиспользуем **уже посчитанные в Радаре `ref`/`volume`** (после ревизии 1):

- `avg_price` (медиана: либо `MarketStatistics.avg_price_24h` для бакета без
  фильтра, либо медиана `SalesHistory` за 7д для бакета с фильтром) — это и
  есть кандидат на `ref` для `make_sell_options`.
- `sales_volume` (соответственно `sales_volume_24h` либо `len(prices)` за 7д) —
  кандидат на `volume_7d`-аргумент `make_sell_options`.

Если `avg_price` есть (не `None`) — считаем `sell_options =
make_sell_options(int(avg_price), sales_volume)` один раз на бакет, затем
проходим по физическим лотам снэпшота и для каждого вызываем
`evaluate_lot_profit`. Если `avg_price is None` (нет ориентира цены — низкое
покрытие `additional_info` qlt/ptn, см. ревизию 1, п.1) — `profitable_offers_count
= None`, поле «нет данных», как и для `avg_price_24h`/`sales_volume_24h`. Это
тот же принцип «не баг», что уже зафиксирован в ревизии 1 — не вычисляем
выгодность без ориентира цены, не подставляем суррогат.

**Что не делаем:** не вызываем `compute_signals_for_entry` и не создаём вторую
параллельную ref-логику — она уже посчитана для других полей той же строки
топа, повторный расчёт того же числа дважды в одном ответе был бы
избыточностью, противоречащей Simplicity First.

### 2. Источник снэпшотов лотов по регионам

`CollectedData` хранится per `(item_id, region)`, причём `region` — поле
`UserWatchlist`, не входит в `GROUP BY` топа (ревизия 1, п.2: топ агрегируется
по `item_id, quality_filter, enchant_filter` без региона). Значит для бакета
нужен отдельный подзапрос регионов.

**Решение:** регионы берём по `item_id` бакета **без** фильтра по
`quality_filter`/`enchant_filter` watcher'а: `SELECT DISTINCT region FROM
user_watchlist WHERE item_id = :item_id AND is_active = true`. Для каждого
найденного региона — последний глобальный снэпшот: `SELECT * FROM
collected_data WHERE item_id = :item_id AND region = :region AND user_id IS
NULL ORDER BY collect_time DESC LIMIT 1`.

**Почему по `item_id`, не по `(item_id, quality_filter, enchant_filter)`:**
`CollectedData.raw_lots` — это сырые лоты аукциона по `item_id + region`, они
физически не зависят от того, с каким `quality_filter` их отслеживает
конкретный watcher (фильтр — атрибут наблюдателя, не лота). Снэпшот один и тот
же для всех бакетов одного `item_id`, независимо от региона watcher'а,
который его инициировал, поэтому собирать регионы с учётом quality/enchant
фильтра было бы лишним усложнением без выигрыша: тот же снэпшот региона будет
переиспользован, даже если регион пришёл от watcher'а с другим фильтром.
Фильтрация по `quality_filter`/`enchant_filter` бакета применяется **после**
получения снэпшота — на уровне отдельных лотов внутри `raw_lots`, через
`_lot_quality_enchant` (см. п.3).

Если у `item_id` несколько регионов среди активных watcher'ов (например EU и
RU) — берём снэпшот по каждому, суммируем число выгодных лотов across
регионов (агрегация по всем регионам вместе — тот же принцип, что уже принят
в ревизии 1, п.2, для цены/объёма).

### 3. Фильтрация лотов по quality/enchant — вынос `_lot_quality_enchant`

`_lot_quality_enchant(lot, master, is_art)` (`backend/app/services/
profitable_lots.py:44-57`) сейчас приватная функция модуля
`profitable_lots.py`. Нужна в `market_radar.py` для фильтрации `raw_lots`
бакета по `quality_filter`/`enchant_filter`, заданным в строке топа — без
неё нельзя корректно сопоставить лот с конкретным бакетом (один и тот же
снэпшот используется несколькими бакетами одного `item_id` с разными
фильтрами).

**Решение (по аналогии с переносом `_build_sales_filter` в ревизии 1):**
перенести `_lot_quality_enchant` (вместе с зависимой от неё мелкой логикой
`_is_artefact`, `_is_liquid` — обе короткие чистые функции без побочных
эффектов) в `backend/app/services/analytics/pricing.py`, рядом с
`_build_sales_filter`, и обновить импорт в `profitable_lots.py` на обратный
(`from app.services.analytics.pricing import _lot_quality_enchant, _is_artefact,
_is_liquid`). Та же причина, что и для `_build_sales_filter`: общий хелпер
оценки лотов должен жить в общем сервисном слое, а не в модуле, заточенном
на одну watchlist-запись пользователя — иначе `market_radar.py` импортирует
из `profitable_lots.py`, что создаёт неочевидную взаимную связь между двумя
независимыми фичами (персональные сигналы и community-агрегация) через
приватные (`_`-префикс) функции одной из них.

**Нужен `master` (объект `MasterItem`)** — уже загружается в
`_calculate_market_radar_aggregate` через `items_by_id` (п.2 исходного
SELECT), переиспользуем без дополнительного запроса.

### 4. Отключение risk-множителя — явный параметр, не магическое значение

Сигнатура `evaluate_lot_profit(buyout_per_unit, amount, sell_options, risk,
min_margin_pct=0.0, batch_stats=None)` (`backend/app/services/analytics/
pricing.py:193`) использует `risk` только в одном месте: `required_margin =
min_margin_pct * RISK_MARGIN_MULT.get(risk, 1.0)` (строка 240). При
`min_margin_pct=0.0` (канонический порог, подтверждённый пользователем)
`required_margin` равен `0` независимо от множителя — risk математически не
влияет на результат при пороге 0%.

**Решение: передаём `risk="low"` явно, не полагаемся на побочный эффект
порога 0%.** `RISK_MARGIN_MULT["low"] = 1.0` — это семантически верный «явно
не повышенный порог», в отличие от передачи произвольной строки (например
`"none"`), которая получила бы тот же множитель `1.0` только через
`.get(risk, 1.0)` фоллбек — случайно, а не по объявленному контракту функции.
`risk="low"` — задокументированное значение домена (`classify_risk` возвращает
одно из `low/medium/high`), поэтому корректно и устойчиво к будущим
изменениям `RISK_MARGIN_MULT` (если когда-нибудь `low` перестанет быть `1.0`,
поведение Радара останется явным и предсказуемым, а не тихо разойдётся с
ожиданием «без риска»).

**Итоговый вызов на каждый лот бакета:**
```python
evaluate_lot_profit(
    buyout_per_unit, amount, sell_options,
    risk="low", min_margin_pct=0.0,
)
```
`batch_stats` не передаём (`None`) — пачковая корректировка цены продажи
(`MarketStatistics.batch_stats`) специфична для конкретного предмета и не
обязательна для грубой агрегатной оценки community-уровня; добавление было
бы лишним запросом `MarketStatistics` сверх уже сделанного в п.1 (там читаем
только `avg_price_24h`/`sales_volume_24h`, не `batch_stats`) — расширять
SELECT ради этого следующая задача, если понадобится точность, не сейчас
(YAGNI).

### 5. Подсчёт `profitable_offers_count`

```python
profitable_offers_count = sum(
    1
    for region_snap in snapshots_for_item  # по всем регионам item_id
    for lot in (region_snap.raw_lots or [])
    if _passes_quality_enchant_filter(lot, master, is_art, quality_filter, enchant_filter)
    and evaluate_lot_profit(
        buyout_per_unit=lot["buyoutPrice"] // lot["amount"],
        amount=lot["amount"],
        sell_options=sell_options,  # посчитаны один раз на бакет, см. п.1
        risk="low",
        min_margin_pct=0.0,
    ) is not None
)
```
(Псевдокод — реализация на стороне `backend-dev`; повторяет фильтры
`buyout <= 0`/`amount <= 0`/ликвидность из `compute_signals_for_entry`, строки
199-206, чтобы не считать заведомо мусорные лоты выгодными.)

Это считается **по уникальным физическим лотам в снэпшоте**, не по
пользователям — 10 watcher'ов, отслеживающих один и тот же бакет, видят то
же самое число `profitable_offers_count`, что и 1 watcher (дедупликация
достигается естественно: лоты снэпшота не привязаны к конкретному
пользователю, бакет проходится один раз независимо от `watchers_count`).

### 6. Производительность и кэш — TTL остаётся 60с, без новой Celery-задачи

**Оценка стоимости:** на каждый из топ-20 бакетов — до нескольких снэпшотов
(по числу уникальных регионов среди watcher'ов конкретного `item_id`,
реалистично 1-3 на текущем масштабе проекта, не 8+), каждый — проход по
`raw_lots` (десятки-сотни лотов на предмет, не тысячи — ограничение самого
аукциона Stalcraft на размещение лотов). Итого худший случай — порядка
20 × 3 × несколько сотен ~ низкие тысячи итераций `evaluate_lot_profit` (чистая
функция без I/O) на один cache-miss, плюс до 60 дополнительных SQL-запросов
снэпшотов (20 бакетов × до 3 региона) сверх уже существующих в п.1/п.2
ревизии 1. Это **дороже текущего**, но не на порядки — `evaluate_lot_profit`
не делает I/O, а SQL-запросы снэпшотов — по первичному ключу/индексу
`ix_collected_time`/`ix_collected_user_item`, не full-scan.

**Решение: оставляем on-the-fly + Redis TTL=60с, без новой Celery-задачи.**
Та же логика, что в исходном архитектурном решении (раздел 2 основного ТЗ):
рост стоимости одного cache-miss не меняет частоту пересчёта (раз в 60с
максимум при активном трафике на страницу) и не обращается к Stalcraft API
вообще — это not рост нагрузки на внешний rate limit, только на собственную
БД/CPU backend-процесса. Один cache-miss раз в 60 секунд, оцениваемый в
низкие тысячи быстрых Python-операций — не уровень, на котором оправдана
отдельная Celery-задача и материализованная таблица (variant B из исходного
ТЗ остаётся отклонённым по той же причине: новая периодическая нагрузка и
точка отказа ради запроса, не бьющего по внешнему API).

**Что не делаем:** не увеличиваем TTL сверх 60с «на всякий случай» — нет
данных, что текущий TTL создаёт проблему; если реальная нагрузка на проде
покажет иное (например, p95 времени ответа `GET /market-radar/` после
ревизии 2 ощутимо деградирует) — это повод вернуться к вопросу постфактум
(YAGNI), не предполагать заранее.

### 7. Имя поля в ответе API

**`profitable_offers_count: int | None`** — добавляется в `MarketRadarItem`
рядом с `watchers_count`/`new_watchers_24h`. `None` — тот же смысл, что
`avg_price_24h is None`: нет ориентира цены для оценки (низкое покрытие
qlt/ptn в `SalesHistory` для бакетов с фильтром, см. ревизию 1 п.1). Название
явно отличается от `watchers_count`, чтобы не создавать впечатление, что это
синонимичные метрики — `watchers_count` про людей, `profitable_offers_count`
про лоты.

### Открытые вопросы

1. **Глубина повторного использования фильтров «мусорных» лотов.** Псевдокод
   п.5 предполагает повтор тех же базовых фильтров, что в
   `compute_signals_for_entry` (`buyout > 0`, `amount > 0`, ликвидность через
   `_is_liquid`, минимальный остаток времени аукциона). Это решение «по
   аналогии», не строго предписанное пользователем — если для community-агрегата
   приемлема более грубая оценка (например, без проверки ликвидности
   `_is_liquid`, считать вообще все лоты со снэпшота) — нужно подтвердить.
   Текущий дефолт (повторить полный набор фильтров) выбран как более
   консервативный и согласованный с существующей логикой, не для
   переосмысления без явного запроса.
2. **Поведение при отсутствии снэпшота вообще** (например, предмет только
   что добавлен в watchlist, коллектор ещё не успел собрать первый снэпшот
   для региона). `profitable_offers_count` в этом случае — `None` (нет
   данных), как и при отсутствии `avg_price`. Не считаем это отдельным
   кейсом, требующим особого UI — тот же «нет данных», что уже принят для
   `avg_price_24h`. Если нужна визуальная дифференциация «нет цены» vs «нет
   снэпшота лотов» — отдельное уточнение, не блокирует реализацию текущего
   дефолта.
