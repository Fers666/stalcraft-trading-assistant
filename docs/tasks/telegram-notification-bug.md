# ТЗ: Расследование бага Telegram-уведомлений (задержка 15-20 мин / не приходят / приходят выборочно)

## Контекст

Пользователь сообщил три симптома одного и того же уведомительного механизма:
1. Уведомления о выгодных лотах приходят с опозданием 15-20 минут.
2. Уведомления иногда не приходят вообще.
3. Из двух пользователей, следящих за одним и тем же лотом, уведомление получает
   только один.

Это исследовательская задача (без реализации фикса). Недавно (2026-07-02,
коммит `9f8086e`) уже чинили смежный баг в том же коде — "6-часовые спайки CPU
— утечка redis-соединений и перекрытие воркеров", корень которого был в
`_publish_signals` (см. ниже). Тот фикс устранял накопление объектов в GC и
даты запуска стат в отдельной цепочке, но НЕ трогал архитектурные причины,
описанные в этом ТЗ — они шире.

## Архитектура (как есть)

Уведомления — это НЕ Celery-таска (`scan_and_notify` намеренно отключён,
см. `backend/app/tasks/celery_app.py:57-58`, комментарий "обрабатываются
telegram_bot сервисом (polling)"). Реальный пайплайн:

```
Celery beat "collect-active-lots" (каждые 20с)
  → collect_all_active_lots() [backend/app/tasks/collectors.py:38]
    → _collect_lots_for_item() [:287]  — по одному (item_id, region) за раз, последовательно
      → StalcraftClient.get_auction_lots() [services/collector/client.py:53]
        → rate_limiter.acquire() [core/rate_limiter.py:103]  — глобальный token bucket в Redis
      → сохраняет CollectedData снэпшот
      → _publish_signals() [:373]  — для КАЖДОГО watchlist-entry с этим item_id/region
                                      (across всех пользователей) вызывает
        → compute_signals_for_entry() [services/profitable_lots.py:53]
        → redis.setex("signals:{user_id}:{item_id}:{region}:{qlt}:{ptn}", TTL=300с, result)

Отдельный контейнер telegram_bot (python-telegram-bot polling, НЕ Celery):
  → _notifier_loop() [telegram_bot/bot.py:250]  — каждые 15с (POLL_INTERVAL)
    → notify_profitable_lots() [:144]
      → для каждого пользователя с telegram_chat_id + notify_telegram + tier.telegram_notifications:
        → для каждой watchlist-записи: redis.get("signals:...")
          → для каждого лота в сигнале: dedup redis.exists("tg_sent:{user_id}:{item}:{region}:{qlt}:{ptn}:{start_time}")
            → app.bot.send_message(...)
            → redis.setex("tg_sent:...", TTL=48ч)
```

Ключевой факт: сигналы и дедупликация уведомлений **per-user** (ключи включают
`user_id`), не глобальные на лот — это устраняет одну из гипотез (см. ниже,
диагноз §4).

## Диагноз

### Причина A (основная, объясняет задержку 15-20 мин И полное отсутствие уведомлений): блокирующий sleep(60) на 429 внутри последовательного цикла коллектора

`backend/app/services/collector/client.py:45-48`:
```python
if response.status_code == 429:
    logger.error("429 received despite token bucket — backing off 60s")
    await asyncio.sleep(60)
    raise RuntimeError("Rate limit exceeded (429)")
```

`collect_all_active_lots()` (`collectors.py:96-105`) обрабатывает пары
`(item_id, region)` **последовательно** внутри одного запуска таски (не
конкурентно — только `await asyncio.sleep(LOTS_REQUEST_DELAY)` между
итерациями). Если ЛЮБОЙ из до 50 (`MAX_LOTS_PER_RUN`, `collectors.py:16`)
предметов в батче получает 429 от Stalcraft API, вся таска блокируется на 60
секунд — это в 3 раза больше, чем интервал самого расписания (`schedule:
timedelta(seconds=20)`, `celery_app.py:31`). Все последующие предметы в этом
батче не собираются ещё 60+ секунд, а Celery beat тем временем продолжает
ставить в очередь `collector` новые экземпляры задачи каждые 20с — при
`worker_concurrency=2` / `-c 2` (`celery_app.py:24`, `docker-compose.prod.yml:60`)
и общей очереди `collector` для ВСЕХ тасок (`collect-active-lots`,
`collect-emission` каждые 120с, `collect-history-and-stats` раз в час) —
задачи начинают копиться, обработка конкретных `(item_id, region)` пар
откладывается на неопределённое время.

Логгирующая фраза "despite token bucket" показывает, что 429 **реально
случается в проде** несмотря на то, что `rate_limiter.acquire()`
(`core/rate_limiter.py:103-137`) уже должен был дождаться доступных токенов
ДО отправки запроса. Это значит, что либо (a) bucket рассинхронизирован с
реальным лимитом Stalcraft (границы минутного окна не совпадают день-в-день:
наш bucket — rolling refill, у Stalcraft может быть fixed calendar-minute
окно), либо (b) какой-то вызов Stalcraft API идёт в обход `rate_limiter`
(нужно грепнуть все места использования `StalcraftClient`/`httpx` напрямую —
в рамках этого расследования не найдено ни одного обхода в текущем дереве
`backend/app`, но не проверялись ad-hoc/ручные скрипты и admin-эндпоинты).

### Причина B (усиливает A, отдельно объясняет полное отсутствие уведомлений даже без 429): TTL сигналов (300с) короче, чем может быть реальный цикл обновления при бэклоге

`profitable_lots.py:21`: `SIGNALS_TTL = 300` — сигнал в Redis живёт 5 минут.
Расчёт батча (`collectors.py:78-85`):
```python
runs_per_cycle = TARGET_CYCLE_SEC / LOTS_REFRESH_INTERVAL  # 60/20 = 3
dynamic_batch = max(MIN_LOTS_PER_RUN, min(ceil(len(due_pairs)/runs_per_cycle), MAX_LOTS_PER_RUN))
```
При `MAX_LOTS_PER_RUN=50` полный цикл обновления всех уникальных
`(item_id, region)` пар масштабируется как `due_pairs / 50 * 20`с. Если
уникальных пар в watchlist (across всех юзеров) больше ~750, полный цикл уже
превышает 300с TTL — сигнал успевает истечь до следующего обновления, и
`notify_profitable_lots()` просто получает `raw = None` и **молча
пропускает** лот (`bot.py:186-187`, `continue` без логирования) — то есть
уведомление не отправляется и не откладывается, оно теряется целиком для
этого цикла обнаружения. Если предмет разово попадает в выгодную зону между
двумя редкими обновлениями — уведомление не придёт вообще, либо придёт с
опозданием на несколько циклов (что и даёт наблюдаемые 15-20 минут — это
несколько пропущенных TTL-окон подряд при бэклоге из-за причины A).

**Требует проверки перед фиксом:** текущее число уникальных активных
`(item_id, region)` пар в `user_watchlist` на проде —
`SELECT COUNT(DISTINCT (item_id, region)) FROM user_watchlist WHERE is_active = true;`
Без этого числа нельзя оценить, насколько велик реальный дрейф цикла.

### Причина C (не баг, вероятная причина «выборочности» №1 — требует подтверждения у пользователя): гейтинг уведомлений по тарифу

`telegram_bot/bot.py:166`:
```python
(user.is_admin or get_tier_limits(user).telegram_notifications)
```
`backend/app/core/tiers.py:26`: тариф `base` (бесплатный) имеет
`telegram_notifications=False` — пользователи на `base` тарифе **никогда** не
получают Telegram-уведомления, независимо от вотчлиста и сигналов. Если один
из двух пожаловавшихся пользователей на `base`, а второй на платном тарифе —
это ожидаемое поведение по продуктовой логике (Phase 0 роадмапа подписок,
`docs/BUSINESS_LOGIC.md` §17), а не баг. **Нужно уточнить у пользователя,
тарифы обоих пострадавших аккаунтов** — если оба на платных тарифах, то это
не причина, и переходим к причине D.

### Причина D (не баг, вероятная причина «выборочности» №2): персональные настройки фильтрации прибыли

Даже при полностью идентичном лоте, `compute_signals_for_entry()` вычисляется
**отдельно на каждую watchlist-запись** с персональными
`min_profit_margin_percent` / `exclude_less_than_amount`
(`UserSettings`, `collectors.py:420-423`, `profitable_lots.py` параметры
`min_profit_margin_pct`/`exclude_less_than_amount`). Два пользователя,
следящих за одним предметом с одинаковыми `quality_filter`/`enchant_filter`,
но разными порогами `min_profit_margin_percent` в Настройках, легитимно
получат разный список "выгодных" лотов — один лот может пройти фильтр
прибыли у одного и не пройти у другого. Это не баг дедупликации (она
per-user и работает корректно, см. ниже), а следствие персонализации.
**Нужно уточнить настройки прибыли обоих пользователей.**

### Причина E (проверена и ИСКЛЮЧЕНА): гипотеза «глобальный dedup на лот» и «один сбойный send рвёт цикл для остальных»

Обе гипотезы из первоначального брифа не подтвердились чтением кода:
- Dedup-ключ `tg_sent:{user.id}:{entry.item_id}:{entry.region}:{quality_filter}:{enchant_filter}:{start_time}`
  (`bot.py:203-207`) включает `user.id` — дедупликация per-user, не глобальная
  на лот. Один пользователь не может "перехватить" уведомление у другого.
- Отправка `app.bot.send_message()` обёрнута в `try/except` **на уровне
  каждого лота** (`bot.py:233-247`), а `dedup`/лог пишутся только при успехе.
  Сбой отправки одному пользователю не прерывает цикл — `continue`
  происходит естественно (это последняя операция в теле цикла), следующий
  лот/пользователь обрабатывается нормально.

### Причина F (потенциальный вклад в задержку, не подтверждён числами): необёрнутый DB-запрос внутри цикла уведомлений — единая на весь batch-сессия

`notify_profitable_lots()` (`bot.py:153-247`) держит **одну** `AsyncSession`
на ВСЕХ пользователей и ВСЕ их watchlist-записи за один проход цикла. Запрос
`select(MasterItem)` (`bot.py:212-215`), выполняемый ради имени предмета для
каждого лота, **не обёрнут в try/except** (в отличие от `json.loads(raw)`,
который обёрнут, `bot.py:189-192`, и в отличие от `send_message`). Если этот
запрос по любой причине бросит необработанное исключение (транзиентный сбой
БД, разрыв соединения) для записи пользователя А в середине списка
`users_to_notify`, исключение всплывает до `_notifier_loop()` (`bot.py:254-257`),
весь текущий 15-секундный цикл прерывается, и пользователи Б, В, ... идущие
ПОСЛЕ пользователя А в порядке итерации (порядок — произвольный порядок
возврата SQL, не гарантированно стабильный), **не обрабатываются в этом
цикле вообще**. Цикл в целом устойчив (следующая итерация через 15с
подхватит всех заново), но при совпадении с причиной B (TTL истёк) может
давать "то приходит, то нет" для конкретной пары пользователь+лот в
конкретный момент. Нельзя подтвердить без логов прод-инстанса
`telegram_bot` (искать `"Notifier loop error"` в логах контейнера) — требует
проверки логов на проде перед фиксом.

### Причина G (второстепенная, не первопричина, но усугубляет A/B): избыточное создание redis-соединений на каждый вызов

И `_publish_signals()` (`collectors.py:427`: `aioredis.from_url(...)` внутри
цикла по item_id/region, до 50 раз за 20с), и `rate_limiter.acquire()`
(`core/rate_limiter.py:112`: то же самое на каждый API-запрос) создают новое
TCP-соединение к Redis на каждый вызов вместо переиспользования пула — это
архитектурное решение, задокументированное как намеренное (docstring в
`rate_limiter.py:18-23`, из-за особенностей `asyncio.new_event_loop()` в
Celery). Прошлый фикс (`9f8086e`) убрал накопление объектов в GC через drain
перед `loop.close()`, но не убрал сам паттерн "новое соединение на каждый
вызов" — под нагрузкой (до 50 предметов × 2+ redis-вызова за 20-секундный
батч) это добавляет заметную латентность на каждый предмет, что усиливает
дрейф цикла из причины B.

## Приоритет причин по вероятности (без данных с прод-БД/логов)

1. **A** (блокирующий 429-sleep) — наиболее вероятная причина задержек
   15-20 мин и части случаев "не пришло вообще". Высокая уверенность, видно
   из кода однозначно.
2. **B** (TTL 300с короче цикла при бэклоге) — прямое следствие A, но может
   происходить и без 429 при органическом росте watchlist. Требует
   подтверждения числом `due_pairs` на проде.
3. **C/D** (тариф/персональные настройки) — наиболее вероятное объяснение
   "выборочности", но это НЕ баг, а вопрос ожиданий пользователя. Требует
   уточнения у пользователя: тарифы и настройки прибыли обоих аккаунтов.
4. **F** — вероятный, но неподтверждённый без логов прод-контейнера
   `telegram_bot`.
5. **G** — второстепенный усилитель A/B, не первопричина сама по себе.

## План исправления по каждой причине (для последующей реализации, НЕ выполнять сейчас)

### Причина A — убрать блокирующий sleep(60) из `_request()` (УТОЧНЕНО после ревью сеньором)

Важное уточнение к исходному диагнозу: внешний `self.retry(exc=exc,
countdown=60)` на уровне таски (`collectors.py:115-116`) **никогда не
срабатывает** для этой ошибки — `RuntimeError` из `_request()` гасится РАНЬШЕ,
внутри `try/except` на уровне отдельного item (`collectors.py:97-103`),
который логирует ошибку и переходит к следующему item. Естественный retry
уже существует окольным путём: `last_successful_check` не обновляется для
упавшего item (строка 108-110), поэтому он естественно попадёт в `due_pairs`
на следующем 20-секундном тике планировщика. Никакой дополнительной
retry-логики городить не нужно.

**Фикс (простой, минимальный риск):**
- В `client.py:45-48` убрать `await asyncio.sleep(60)`, оставить немедленный
  `raise RuntimeError("Rate limit exceeded (429)")`.
- Ничего в `collectors.py` менять не нужно — существующий per-item
  try/except и механизм `due_pairs` уже корректно обрабатывают retry.
- Добавить лог "сколько раз в час происходит 429 despite token bucket" (см.
  причину B ниже) для последующей диагностики рассинхрона bucket — это
  ТОЛЬКО логирование, не меняет throughput/лимиты.
- НЕ трогать `CAPACITY`/`REFILL_RATE` в этой итерации — рассинхрон bucket
  (rolling vs calendar minute) остаётся открытым вопросом, требующим
  прод-логов, не блокирует этот фикс.

### Причина B — ТОЛЬКО добавить наблюдаемость в этой итерации, НЕ менять лимиты

Решение пользователя (2026-07-06): в этой итерации менять `SIGNALS_TTL` /
`MAX_LOTS_PER_RUN` / параметры rate limiter НЕ будем — недостаточно
прод-данных. Делаем только:
- Лог факта и частоты 429 "despite token bucket" (переиспользует лог из
  причины A, уже туда добавлен).
- Лог/метрику фактической длины полного цикла обновления watchlist (время
  между двумя последовательными `due_pairs` проходами по одной и той же
  паре item/region) — чтобы в следующий раз при жалобе на задержки сразу
  было видно число без ручного SQL по проду.
- Дальнейшие шаги (менять TTL/batch) — только после того, как эти логи
  накопятся и покажут реальный дрейф цикла на проде.

### Причина C/D — не фикс, а UX-уточнение

- Если после уточнения у пользователя выяснится, что оба аккаунта на
  платных тарифах с одинаковыми persoanl-настройками прибыли — тогда это
  всё же баг, и придётся углубиться в конкретные watchlist-записи этих двух
  пользователей (сравнить `quality_filter`/`enchant_filter`/`UserSettings`
  построчно). Если тариф/настройки объясняют разницу — фикса не требуется,
  предложить добавить пользователю на фронте пояснение (например, в
  Настройках → Telegram, если тариф не поддерживает уведомления).

### Причина F — обернуть весь цикл на пользователя в try/except

- В `notify_profitable_lots()` обернуть тело `for entry in watchlist:`
  (или хотя бы запрос `MasterItem`) в try/except с логированием per-entry
  ошибки, аналогично тому, как уже сделано для `json.loads` и `send_message`
  — чтобы сбой на одной записи одного пользователя не прерывал обработку
  остальных пользователей в этом же цикле.
- Проверить прод-логи контейнера `telegram_bot` на предмет строки
  `"Notifier loop error"` — если она встречается регулярно, это прямое
  подтверждение причины F.

### Причина G — переиспользование Redis-соединения (ПОДТВЕРЖДЕНО сеньором, включаем в этот фикс)

Обоснование "новое соединение на каждый вызов из-за нового event loop на
каждую Celery-таску" оправдывает пересоздание соединения МЕЖДУ тасками, но
не оправдывает пересоздание ВНУТРИ одного запуска батча (до 50 items ×
минимум 2 redis-вызова = 100+ TCP-хендшейков за 10-20с). Это отдельная,
более узкая граница, которую можно оптимизировать без риска для
межтасковой изоляции.

**Фикс:**
- В `collect_all_active_lots._run()` создать ОДИН `aioredis` клиент в
  начале `_run()` (в текущем event loop, созданном `run_async()`), передать
  его явным параметром в `_collect_lots_for_item` → `_publish_signals` и в
  `rate_limiter.acquire(cost=..., redis_client=...)` (добавить опциональный
  параметр, по умолчанию — создавать новый, чтобы не ломать другие вызовы
  `rate_limiter` вне этого батча, например `collect_all_history`).
  Закрыть клиент (`aclose()`) в конце `_run()`.
- Явно НЕ трогать глобальный синглтон/кеширование между тасками — только
  внутрибатчевое переиспользование.

## Открытые вопросы / требует подтверждения

- Тарифы (`user.tier`) и персональные `UserSettings.min_profit_margin_percent`
  / `exclude_less_than_amount` двух пострадавших пользователей — без этого
  нельзя разделить причины C/D (не баг) от реального бага в дедупликации.
- Число уникальных `(item_id, region)` в активном watchlist на проде — нужно
  для количественной оценки причины B (запрос дан выше, требуется выполнение
  на проде, не локально).
- Частота 429-ошибок в логах worker/scheduler на проде за последние дни —
  нужно для подтверждения причины A и понимания, действительно ли bucket
  рассинхронизирован с реальным лимитом Stalcraft.
- Логи контейнера `telegram_bot` (`"Notifier loop error"`) — нужно для
  подтверждения причины F.
- Любое изменение `MAX_LOTS_PER_RUN` / `SIGNALS_TTL` / параметров
  `TokenBucketRateLimiter` (`CAPACITY`/`REFILL_RATE`) — по правилам проекта
  требует явного подтверждения пользователя перед реализацией, так как
  напрямую влияет на частоту опроса Stalcraft API.
