# ТЗ: Расследование повторяющихся CPU-спайков на проде (после фикса 2026-06-29)

## Контекст

2026-06-29 (коммит `9f8086e`, `docs/CHANGELOG.md`) уже чинили инцидент
"CPU-спайки раз в час" — тогда причиной был `collect_all_history`,
обрабатывавший watchlist строго последовательно (50.7с на прогон),
пересекаясь по времени с `collect_all_active_lots` на обоих forked
worker-процессах (`worker_concurrency=2`). Фикс: `HISTORY_CONCURRENCY=6`
(round-robin чанки через `asyncio.gather`) + перенос
`calculate_all_market_stats` из отдельного `crontab(minute=5)` в цепочку
`.delay()` сразу после `collect_all_history`. QA локально показал 6.3с вместо
50.7с.

2026-07-06 пользователь прислал два скриншота Grafana с прода и сообщил
«ранее делали эту задачу, не помогло». Графики показывают ИЗМЕНИВШИЙСЯ, а не
идентичный паттерн:

1. За 12ч (06:40–18:40): ~10-11 пиков Core_0/Core_1 до 60-70% (один почти
   100%), примерно раз в час, фон между пиками ~15-20%.
2. Zoom на последний час (17:41–18:41): НЕ короткий спайк, а два разных
   события —
   - **Плато 18:01–18:11 (~10 мин):** Core_0 почти 100%, держится, не
     мгновенный всплеск; Core_1 колеблется 50-70%.
   - **Отдельный острый скачок ~18:33 (пара минут),** через ~20+ минут ПОСЛЕ
     конца плато, не совпадающий явно ни с одним cron-расписанием.

Это исследовательская задача (без реализации фикса). Есть параллельное,
незакоммиченное расследование `docs/tasks/telegram-notification-bug.md`
(причины A/G — блокирующий `sleep(60)` на 429 и лишние Redis-соединения) —
оно покрывает частично смежный код (`collectors.py`, `client.py`,
`rate_limiter.py`), но НЕ объясняет 10-минутное плато и отложенный скачок
через 20 минут напрямую. Пересечение упомянуто ниже (см. §"Связь с причиной
A"), но не является главным диагнозом здесь.

## Диагноз

### Ключевой факт из кода: фикс 06-29 распараллелил ТОЛЬКО `collect_all_history`, но НЕ `calculate_all_market_stats`

`backend/app/tasks/analyzers.py::calculate_all_market_stats` (строки 38-79)
осталась **строго последовательной**: `for item_id, region in unique_pairs:
await calculate_market_stats(...)` — один Python-процесс/один event loop,
без `asyncio.gather`, без чанков, без какой-либо конкурентности. Это НЕ было
затронуто коммитом `9f8086e` — диф того коммита (см. `git show 9f8086e`)
трогал только `run_async()` (drain GC) и переносил вызов из отдельного cron в
цепочку `.delay()` после `collect_all_history`. Сама функция
`calculate_all_market_stats` не менялась.

### `calculate_market_stats` (`backend/app/services/analytics/market_stats.py`) — тяжёлая CPU-bound операция на предмет

Для КАЖДОЙ уникальной пары `(item_id, region)`:
- `SELECT ... SalesHistory WHERE sale_time >= now-30d` — полная ORM-загрузка
  (`.scalars().all()`) всех продаж за 30 дней, без лимита строк. Для
  популярных предметов это могут быть тысячи строк.
- Минимум 4-5 полных проходов по `sales_30d` в чистом Python (не SQL
  agregate): `safe_stats()` ×3 (24ч/48ч/7д), группировка `by_hour`/`by_day`/
  `by_day_hour` (вложенные словари, `weighted_score()` пересчитывает
  `statistics.mean` для каждой группы), `_calculate_batch_stats()` (ещё один
  полный проход + 6 сегментов по `amount`), `_calculate_sell_options()`
  (ещё один проход + доп. `SELECT` снэпшотов), `_avg_sell_time_from_buyouts()`,
  `_recent_bulk_signal()`.
- Всё это выполняется **последовательно по предметам, в одном процессе, на
  одном ядре** — классический CPU-bound Python-код (`statistics.mean/stdev`,
  списковые включения, сортировки), не I/O-bound. В отличие от
  `collect_all_history` (в основном I/O: сетевые запросы к Stalcraft API +
  ожидание БД), где параллелизация через `asyncio.gather` реально помогает
  (несколько ожиданий I/O перекрываются), здесь параллелизация корутинами
  внутри ОДНОГО процесса почти не даст выигрыша для чистых вычислений (GIL/
  кооперативный event loop на одном ядре) — этот код не распараллелен вообще.

### Почему характер изменился именно сейчас (не в 06-29)

С 06-29 репозиторий получил: раздел новостей (`4d80d17`), emission-трекер
(`5dd64ff`). Emission не использует watchlist (отдельный лёгкий таск раз в
120с, 1 токен) — сам по себе не увеличивает нагрузку `calculate_market_stats`.
Но рост продукта (новые фичи, вероятный рост пользователей/watchlist между
06-29 и 07-06) мог органически увеличить:
1. **Число уникальных `(item_id, region)` пар** в активном watchlist — прямой
   множитель для количества итераций `calculate_all_market_stats` (и для
   `collect_all_history`, хотя тот уже распараллелен на 6 чанков с фиксированным
   потолком, без динамического масштабирования в отличие от
   `collect_all_active_lots`, у которого есть `TARGET_CYCLE_SEC`/`dynamic_batch`).
2. **Объём `SalesHistory` на популярный предмет** — таблица копится 120 дней
   (`will_be_deleted_at`), с 06-29 прошла неделя непрерывного сбора истории
   каждый час — на конец недели каждый активный предмет накопил на порядок
   больше строк `sales_30d`, чем в момент локального QA-теста 06-29 (который,
   вероятно, гонялся на существенно меньшей/тестовой БД). Это ОРГАНИЧЕСКИЙ
   рост стоимости `calculate_market_stats` НЕЗАВИСИМО от роста watchlist —
   таблица растёт каждый час сама по себе.

Оба фактора напрямую растят стоимость именно той функции, которую фикс 06-29
не трогал.

### Гипотеза по двум паттернам на графике

- **Плато 18:01–18:11 (~10 мин)** = `collect_all_history` — уже
  распараллелен (6 чанков), но при выросшем числе уникальных пар суммарное
  время работы всё равно растёт (фиксированный потолок `HISTORY_CONCURRENCY=6`
  не масштабируется с размером watchlist, в отличие от `collect_all_active_lots`).
  Отсюда устойчивое (не мгновенное) плато вместо короткого пика — согласуется
  с тем, что задача стартует в `18:00:00` (crontab `minute="0"`) и разгорается
  к `18:01` на графике.
- **Отдельный скачок ~18:33** = `calculate_all_market_stats` — НЕраспараллеленная,
  чисто CPU-bound операция, которая наконец получила своё место в очереди и
  выполняется одним процессом на одном ядре — короткий, но острый пик
  (в отличие от растянутого I/O-bound плато истории).
- **Разрыв ~20+ минут между ними** — наиболее вероятное объяснение: очередь
  `"collector"` общая для ВСЕХ тасок (`task_routes={"app.tasks.*": {"queue":
  "collector"}}`), и `worker_concurrency=2` — во время 10-минутного плато
  `collect_all_active_lots` (расписание каждые 20с) продолжает планироваться
  Celery beat и копится backlog'ом, пока оба worker-слота заняты
  `collect_all_history`. Когда история завершается и `calculate_all_market_stats.delay()`
  ставится в очередь, ему приходится ждать, пока воркеры разгребут накопленный
  backlog активных лотов — FIFO-семантика брокера (Redis list) не даёт
  приоритета вновь добавленной таске. Разгребание ~30 накопленных инстансов
  `collect_all_active_lots` (10 мин × 3 в минуту) двумя воркерами объясняет
  задержку в районе 20 минут, ОСОБЕННО если часть из них попадает в
  блокирующий `sleep(60)` при 429 (причина A из `telegram-notification-bug.md`,
  уже задокументирована, там же лежит фикс — не реализовывать в рамках этой
  задачи, но стоит учитывать как усилитель именно ЗАДЕРЖКИ между эпизодами,
  не как источник самой CPU-нагрузки: `asyncio.sleep` не грузит CPU, но
  занимает worker-слот, продлевая разгребание backlog).

### Альтернативные гипотезы (менее вероятные, не исключены)

- **Рост `collect_all_history` сам по себе — единственная причина, без вклада
  `calculate_all_market_stats`.** Возможно, но не объясняет ВТОРОЙ отдельный
  скачок через 20 минут — если бы вся нагрузка была в истории, не было бы
  причины для второго изолированного пика именно там, где `calculate_all_market_stats`
  архитектурно обязана быть (следующее звено цепочки).
- **429/rate-limit шторм именно в это время** — сам по себе не создаёт CPU
  нагрузку (сон не ест CPU), но мог совпасть по времени случайно. Не
  подтверждается графиком (плато ровное, не рваное, как было бы при частых
  ошибках/ретраях).
- **`emission` таска (`collect_emission`, каждые 120с)** — слишком лёгкая (1
  токен, минимум вычислений) чтобы дать 100%-й пик на ядро; исключаем как
  первопричину, но она добавляет к общему числу тасок в общей очереди
  (усугубляет тесноту `worker_concurrency=2`, не первопричина).
- **`master_items` фоновый сбор (2236 предметов)** — не реализовано (см.
  `docs/NOTES.md`, раздел "Незадействованные возможности API"), не может быть
  причиной сейчас.

## Приоритет причин по вероятности (без прод-данных)

1. **`calculate_all_market_stats` — нераспараллеленная, CPU-bound, растёт с
   объёмом `sales_history` И с ростом watchlist.** Высокая уверенность из
   чтения кода — единственная функция в цепочке, оставшаяся полностью
   последовательной после фикса 06-29, и единственная объясняющая ВТОРОЙ
   изолированный скачок.
2. **`collect_all_history` при выросшем watchlist — уже распараллелена, но
   `HISTORY_CONCURRENCY=6` не масштабируется динамически.** Вероятный вклад в
   растянутое 10-минутное плато, но само по себе не объясняет второй скачок.
3. **Общая очередь `"collector"` + `worker_concurrency=2` — структурная
   теснота.** Не первопричина CPU сама по себе, но объясняет наблюдаемый
   ЗАЗОР (~20 мин) между плато и скачком через backlog `collect_all_active_lots`.
4. **429/`sleep(60)` (причина A из другого ТЗ) — усиливает задержку разгребания
   backlog, не создаёт CPU-нагрузку напрямую.** Второстепенный, но стоит
   проверить логи на совпадение по времени.

## План (НЕ реализовывать сейчас — только для будущей задачи)

Ранжировано по риску/эффекту, минимальные точечные изменения:

1. **Добавить логирование длительности `calculate_all_market_stats`** (время
   `_run()` целиком + время на пару, аналогично уже добавленному логу цикла
   watchlist в `_collect_lots_for_item`) — чистое наблюдение, нулевой риск,
   даёт число для следующего анализа без догадок.
2. **Профилировать/оптимизировать `calculate_market_stats` на уровне SQL**,
   не Python: перенести часть агрегаций (`avg`/`min`/`max`/`count` за 24ч/48ч/7д/30д)
   в SQL (`func.avg`, `func.count`, `GROUP BY` по временным окнам) вместо
   загрузки всех строк `SalesHistory` ORM-объектами в Python и многократных
   проходов `statistics.mean/stdev` — это уменьшит и сетевой трафик из БД, и
   CPU-стоимость на предмет, без изменения формул (не трогает
   `BUSINESS_LOGIC.md`). Более рискованная правка (нужны тесты на
   идентичность результатов), делать отдельным ТЗ backend-dev с TDD.
3. **Рассмотреть отдельную Celery-очередь/worker для CPU-тяжёлых редких тасок**
   (`collect_all_history` + `calculate_all_market_stats`) отдельно от частых
   лёгких (`collect_all_active_lots`, `collect_emission`) — например, вторая
   `worker`-нода с `-Q analytics -c 1` (доп. риск: требует ресурсов, ещё один
   контейнер). Это НЕ решает CPU-стоимость самой функции, но изолирует
   всплеск от постоянного 20-секундного сбора лотов — устраняет накопление
   backlog и разрыв в 20 минут. Компромисс: на 2-vCPU сервере доп. worker всё
   равно конкурирует за то же физическое CPU — реального выигрыша по
   суммарной загрузке не будет, только по распределению приоритета (активные
   лоты не будут ждать позади истории/статистики). **Требует подтверждения
   пользователя**, так как это архитектурное изменение инфраструктуры
   (`docker-compose.prod.yml`), не просто код.
4. **Ограничить частоту/объём `calculate_all_market_stats`** — например,
   считать статистику не для ВСЕХ уникальных пар каждый час, а только для тех,
   у кого появились новые продажи с последнего расчёта (дифф вместо полного
   пересчёта). Более сложный рефакторинг, потенциально самый эффективный
   долгосрочно, но не минимальный — отдельная задача после профилирования
   (пункт 2), чтобы сначала понять, где реально уходит время.
5. **НЕ менять** `HISTORY_CONCURRENCY`, `worker_concurrency`,
   `MAX_LOTS_PER_RUN`, частоту расписаний — это либо требует отдельного
   подтверждения пользователя (влияет на частоту опроса/rate limit), либо
   преждевременно без чисел из профилирования (пункт 2 выше должен идти
   первым).
6. Держать в уме связь с `docs/tasks/telegram-notification-bug.md` (причина
   A/G) — при реализации фикса для CPU-спайков логично проверить, не решает
   ли уже запланированный там фикс (`asyncio.sleep(60)` → immediate raise,
   общий Redis-клиент на батч) заодно и часть проблемы с длиной backlog в
   очереди `collector`. Не дублировать работу — если тот фикс будет
   реализован раньше, переоценить актуальность разрыва ~20 мин после него.
