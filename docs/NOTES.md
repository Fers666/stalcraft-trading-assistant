## Вопросы к бизнес-логике

- **Пачки (tracked_batch_sizes)** — поле хранится в `user_watchlist` ([10, 20, 30, 50] по умолчанию),
  но **нигде не используется в аналитике**. Алгоритм `find_best_batch_combination` принимает
  `target_quantity` из запроса, а не из watchlist.
  **Варианты решения:**
  1. Убрать поле как лишнее — пользователь сам вводит количество при запросе
  2. Реализовать группировку sell_options по этим размерам (например, "купить пачку 50 шт за X ₽")
  ⚠️ **Нужно решить до реализации страницы Склада.**

## Система апрувов (доступ к сайту)

**Концепция:** закрытый портал — пользователи не могут войти без разрешения администратора.

**Логика:**
- При регистрации пользователь ждёт апрув (статус: pending)
- Администратор через панель выдаёт апрув на указанное количество дней (вводит вручную)
- После истечения срока — доступ закрывается, нужен новый апрув
- Без апрува: авторизация не проходит, даже если аккаунт существует

**Интерфейс admin-панели:**
- Строка поиска пользователей (по имени / email)
- Список пользователей с текущим статусом (pending / active / expired)
- Кнопка "Выдать апрув" → поле ввода количества дней → подтвердить
- Отображение: до когда действует апрув (дата истечения)

**Главная страница (лендинг для незалогиненных):**
- Короткое описание что такое портал и что он делает
- Кнопки "Войти" / "Зарегистрироваться"
- Пояснение что доступ по апруву (чтобы не было путаницы у новых пользователей)

⚠️ **Backend:** добавить поле `approved_until` (DateTime) в таблицу `users` + проверка при логине.
